# UC-DSH-070: Инкрементировать счётчик просмотров блюда

**Version:** 1.0 | **Date:** 2026-06-07

---

## Actors (Инициаторы)

- Primary: клиент (веб-приложение, мобильное приложение) — пингует эндпоинт после успешного рендера карточки блюда. Пользователь может быть как анонимным, так и аутентифицированным.
- Secondary: нет.

---

## Resource (Ресурс)

- Entity: `Dish` — поле `Dish.ViewsCount`.
- Identifier: `Dish.Id` (`Guid`) — path-параметр.
- Action: Increment (атомарный `UPDATE ... SET ViewsCount = ViewsCount + 1`).

---

## Security (Безопасность)

### Authentication (Аутентификация)

None. Эндпоинт публичный (`[AllowAnonymous]`). Опубликованная карточка — публичный ресурс; гостевые просмотры считаются наравне с аутентифицированными.

### Authorization (Авторизация)

- Policy: нет (любой может пинговать просмотр публичного блюда).
- Roles: any (включая анонимных).
- Ownership: не учитывается. Самопросмотр автора в текущей реализации тоже считается — см. раздел «Реализация / Отложено».

### State Constraints (Ограничения по состоянию)

- Инкремент работает **только** для блюд со статусом `DishStatus.Published`. Условие зашито в `WHERE` SQL-запроса — для черновика, снятого с публикации или архивного блюда инкремент не происходит, в ответе — `404`.

### Contextual Constraints (Контекстуальные ограничения)

Нет. Антинакрутка / дедупликация просмотров на текущем этапе не реализована — ответственность на клиенте (один пинг = один реальный рендер).

---

## API Contract (Контракт API)

### Endpoint

```
POST /api/dishes/{id}/views
```

### Request (Запрос)

**Path Parameters:**

- `id` — `Guid` идентификатор блюда.

**Headers:**

- `Authorization: Bearer <JWT>` — опционально (эндпоинт `[AllowAnonymous]`). Если передан, JWT проверяется стандартным middleware, но результат на инкремент не влияет.

**Body:**

Не требуется.

### Response (Ответ)

- Status: `204 No Content`.
- Body: отсутствует.

### Errors (Ошибки)

| HTTP Status | Код ошибки домена  | Условие                                                                                                                                                                                  |
|-------------|--------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 404         | `DISHES.DISH_NOT_FOUND` | Блюдо с указанным `Id` отсутствует **или** не находится в статусе `Published`. В обоих случаях ответ одинаковый, чтобы не раскрывать существование черновика стороннему пользователю. |

---

## Preconditions (Предусловия)

- HTTP-маршрут `POST /api/dishes/{id:guid}/views` корректно резолвится.
- В системе существует блюдо с переданным `Id` и `Status = Published`.

---

## Invariants (Инварианты домена)

- `Dish.ViewsCount >= 0` — поддерживается CHECK-constraint `CK_Dishes_ViewsCountNonNegative` в БД. Инкремент `+1` его не нарушает.
- `Dish.UpdatedAt` **не** изменяется при инкременте просмотров. Это специальное правило `UpdatedAtInterceptor` (поле `ViewsCount` исключено из триггера обновления `UpdatedAt`); UC-070 идёт мимо `ChangeTracker` через `ExecuteUpdateAsync`, поэтому интерсептор тут не задействуется в принципе.
- `Dish.PublishedVersionData` (jsonb-снепшот) **не** содержит `ViewsCount` — это runtime-счётчик, исключён из снепшота. Инкремент не инвалидирует опубликованную версию.

---

## Main Flow (Основной поток)

1. Клиент успешно отрисовал публичную карточку блюда (UC-DSH-050 / UC-DSH-051).
2. Клиент шлёт `POST /api/dishes/{id}/views` без тела.
3. Контроллер `DishesController.IncrementViewsAsync` создаёт `IncrementDishViewsCommand(DishId = id)` и отправляет в MediatR.
4. `IncrementDishViewsCommandHandler` вызывает `IDishRepository.IncrementViewsAsync(dishId, ct)`.
5. Репозиторий выполняет один SQL:
   ```sql
   UPDATE dishes."Dishes"
   SET "ViewsCount" = "ViewsCount" + 1
   WHERE "Id" = @id
     AND "Status" = 1 /* Published */;
   ```
6. PostgreSQL возвращает `1` затронутую строку → Handler возвращает `Result.Success()`.
7. Контроллер возвращает `204 No Content`.

---

## Alternative Flows (Альтернативные потоки)

Нет. У UC только один успешный путь.

---

## Edge Cases (Граничные случаи)

- **EC-1: Блюдо отсутствует.** `WHERE Id = @id` не найдёт строку, `ExecuteUpdateAsync` вернёт `0`. Handler возвращает `DishesErrors.DishNotFound` → `404 DISHES.DISH_NOT_FOUND`. Состояние БД не меняется.

- **EC-2: Блюдо в статусе `Draft` / `Unpublished` / `Archived`.** Условие `Status = Published` в `WHERE` отсекает запись; `ExecuteUpdateAsync` вернёт `0`. Ответ — `404 DISHES.DISH_NOT_FOUND`, идентичный EC-1 (не раскрываем существование черновика стороннему пользователю).

- **EC-3: `Guid.Empty` в path.** ASP.NET route-binding принимает (это валидный Guid); SQL вернёт `0` строк → `404`. Альтернативная реакция (`400 VALIDATION.ERROR`) намеренно не реализована — публичный эндпоинт не должен различать «битый запрос» и «несуществующее блюдо» (защита от probing).

- **EC-4: Конкурентные запросы (несколько пингов одновременно).** `UPDATE ... SET ViewsCount = ViewsCount + 1` атомарен на уровне строки в PostgreSQL (Read Committed снимает row-lock на время инкремента). Race condition невозможен — каждый запрос увеличит счётчик корректно.

- **EC-5: Самопросмотр автора.** На текущем этапе считается наравне с прочими просмотрами. Запись в техдолге будущих этапов — добавить фильтр `AuthorUserId <> @currentUserId`, когда появится бизнес-причина.

- **EC-6: Многократные пинги от одного клиента.** Дедупликация на сервере не реализована. Каждый вызов = `+1`. Антинакрутка — задача будущих этапов (см. техдолг).

---

## Postconditions (Постусловия)

При успешном выполнении:

- `Dish.ViewsCount` увеличен на 1 для блюда с переданным `Id`.
- `Dish.UpdatedAt` не изменён (специальное исключение для счётчиков).
- `Dish.PublishedVersionData` не изменён (jsonb-снепшот не содержит `ViewsCount`).
- Доменные события **не** поднимаются — это runtime-счётчик, не доменное действие.

При неуспехе (`404`):

- Состояние БД не меняется.

---

## Non-Functional (Нефункциональные требования)

- **Idempotency.** **Не идемпотентен** — каждый вызов увеличивает счётчик на 1. Это семантически правильно: один пинг = одно событие просмотра. Идемпотентность на сервере не требуется; повторные пинги от того же клиента — забота клиента.
- **Rate Limit.** Не реализован на Этапе 2. Endpoint публичный, потенциально уязвим к накруткам — общий HTTP-уровневый rate limit и более сложная антифрод-логика отложены в техдолг будущих этапов.
- **Performance.** Целевое < 5 мс. Один `UPDATE` по PK без `SELECT`, без загрузки агрегата, без `SaveChangesAsync` через `ChangeTracker`. Триггеры/интерсепторы не задействуются.
- **Consistency.** Read Committed (стандарт PostgreSQL). Row-level lock на время инкремента — не блокирует другие строки и не конфликтует с чтением карточки.
- **Audit.** Стандартное HTTP-логирование (Serilog). Отдельно факт просмотра не логируется.

## Реализация Этапа 2 — что в наличии и что отложено

### Реализовано

- Атомарный инкремент через `IDishRepository.IncrementViewsAsync` (EF Core 8 `ExecuteUpdateAsync`).
- Зашитое в SQL условие `Status = Published` — единое 404-поведение для несуществующего/черновика/архива.
- Публичный эндпоинт `[AllowAnonymous]`.
- Раздельное от UC-DSH-050/051/052 управление: клиент сам решает, когда пинговать (после рендера, после авторизации, не для пререндера/SSR).

### Отложено

- **Фильтр самопросмотров автора.** Когда появится бизнес-причина — добавить условие `AuthorUserId <> @currentUserId` в `WHERE` (либо короткий `SELECT AuthorUserId` перед инкрементом). Запись в `docs/_private/private_TODO-будущие-этапы.md`.
- **Антинакрутка / дедупликация.** Уникальные просмотры за окно времени (например, через Redis: ключ `dish-view:{userOrIp}:{dishId}`, TTL 30 минут). Отложено вместе с появлением Redis в стеке.
- **Метрика «реальной популярности» при появлении приватных рецептов.** Когда часть рецептов станет доступна только подписчикам (модуль Subscriptions, Этап 8+), сравнение `ViewsCount` между публичными и приватными карточками станет некорректным — гостевой трафик «накрутит» публичные. Возможные стратегии (на будущее): два счётчика `ViewsAnonymous` + `ViewsAuthorized`; уникальные просмотры за день; учёт только авторизованных. Запись в `docs/_private/private_TODO-будущие-этапы.md`.
