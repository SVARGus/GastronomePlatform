## Концепция: UC ↔ API Endpoint ↔ UI Screen

Перед чтением списка UC важно зафиксировать соотношение трёх уровней абстракции:

| Уровень | Что это | Сколько на одну UI-страницу |
|---------|---------|------------------------------|
| **Use Case** | Один логически законченный бизнес-сценарий | Много (5–20) |
| **API Endpoint** | Один HTTP-маршрут (`POST /api/dishes/{id}/steps`) | Много (≈ 1:1 с UC) |
| **UI Screen** | Одна страница в приложении / на сайте | 1 |

**Правила соответствия:**

- **Один UC ↔ один Endpoint** (почти всегда — 1:1).
- **Одна UI-страница ↔ много UC и много Endpoints**.

UI-страница — это **оркестратор**, который вызывает много UC. Например, страница «Создание блюда» (Этап 4) использует:

- `UC-DSH-057 GetCategoryTree` — для селектора категорий
- `UC-DSH-061 GetPopularTags` — для облака тегов
- `UC-DSH-064 GetMeasureUnits` — для единиц измерения
- `UC-DSH-060 SearchTagsAutocomplete` — при вводе тегов
- `UC-DSH-062 SearchIngredients` — при добавлении ингредиента
- `UC-DSH-001 CreateDish` — при первом сохранении
- `UC-DSH-002 UpdateDishCard` — при последующих правках
- `UC-DSH-020 AddRecipeStep` — на каждом добавлении шага
- … и так далее

То, что несколько UC используются на одной UI-странице, **не значит**, что их нужно объединить в один большой UC. Атомарность UC — это нормально и правильно.

### UI-паттерн «Batched Commit» (Этап 4+)

Бэкенд на Этапе 2 поддерживает **атомарные UC** — каждый отдельный вызов сохраняется в БД сразу. UI на Этапе 4+ может реализовать другой пользовательский паттерн: накапливать изменения локально и при нажатии «Сохранить» отправлять их пакетно.

Этот паттерн (и требования к нему — собрать ошибки всех вызовов, показать пользователю списком, обеспечить транзакционность) — задача Этапа 4. На Этапе 2 бэкенд про это не знает: для него каждый запрос остаётся атомарным.

## Концепция: Лицензионная модель пользовательского контента

Принята **«Лицензия безотзывная»** (стандарт индустрии: YouTube, Хабр, VK):

- Автор сохраняет исключительные авторские права на свой контент.
- При публикации автор предоставляет платформе неисключительную, безотзывную, всемирную, безвозмездную лицензию на использование, отображение, воспроизведение и модификацию контента.
- Лицензия не отзывается при снятии блюда с публикации, удалении аккаунта и т.п.

### Что значит для UC

| Действие | Каталог/Поиск | Прямая ссылка | Участники прошлых заказов (Этап 6+) |
|----------|----------------|----------------|--------------------------------------|
| `Unpublish` | Скрыто | 404 / «Снято автором» | **Видят** (через snapshot в Orders) |
| `Archive` | Скрыто | Скрыто | **Видят** (через snapshot в Orders) |
| Hard delete (Этап 8+) | Удалено | Удалено | **Видят** (через snapshot в Orders) |

Модуль `Orders` (Этап 6) при создании заказа сохраняет собственный снепшот блюда (`OrderItem.DishSnapshot jsonb`). Этот снепшот не зависит от состояния `Dish` в модуле Dishes.

### Удаление аккаунта ≠ отзыв контента

- **Удаление аккаунта** (Этап 5+): автор больше не логинится; в UI его блюда показываются с маркером «Автор удалил аккаунт». Сами блюда **остаются опубликованными**.
- **Отзыв контента** (явный): только через UI блюда → «Снять с публикации». Действует поштучно.

## Концепция: Авторизация модификаций (POL-001)

Все UC модификации блюда подчиняются единой политике авторизации **POL-001 Dish Ownership Policy** (см. `POL-001-dish-ownership.md`).

**Краткая формулировка для Этапа 2:**

Действие разрешено если:
- Пользователь является автором блюда (`Dish.AuthorUserId == ActorUserId`), ИЛИ
- Пользователь имеет роль `Admin`.

**С учётом статуса блюда:**
- `Archived` — модифицировать может только `Admin`.
- `Draft` / `Published` / `Unpublished` — Author и Admin (с ограничениями по типам полей).

**Будущие расширения:**
- Этап 4+: добавится проверка `DishOwnership` (PlatformOwned-блюда — только Admin).
- Этап 8+: появятся роли `Moderator` и `Support` (с тикетом).

В UC-описаниях ниже вместо «только автор» / «автор и админ» используется ссылка `Authorization: POL-001`.

## Группировка по подразделам

### 001–099. Пользовательские сценарии

#### Управление блюдом (CRUD + lifecycle)

##### UC-DSH-001 — Создать черновик блюда

**Тип:** Command. **Статус:** Core. **Этап:** 2.

Автор (любая роль кроме Guest) создаёт `Dish` в статусе `Draft`. На входе — Name + минимум обязательных полей. Возвращает Id созданного блюда. Slug генерируется автоматически из Name (с разрешением коллизий через суффикс). Сразу создаётся вложенный пустой `Recipe` с дефолтными `Timing` и `Yield`. Если в команде указан `MainImageId` — выполняется eager attach к Media через `IMediaService`.

##### UC-DSH-002 — Обновить публичную карточку блюда

**Тип:** Command. **Статус:** Core. **Этап:** 2.
**Authorization:** POL-001.

Поля: Name, ShortDescription, Description, DifficultyLevel, CostEstimate, MainImageId, OwnerType. При смене MainImageId — старое медиа отвязывается через `DetachFromEntityAsync`, новое — `AttachToEntityAsync`. Slug **не** меняется автоматически даже при смене Name.

**Правка не трогает `PublishedVersionData`:** изменения сохраняются в основной таблице `Dish`, но публичная версия (если блюдо опубликовано) не обновляется. Чтобы изменения стали видны посетителям — нужен явный `UC-DSH-004 PublishDish`. Поле `Dish.UpdatedAt` обновляется автоматически через `SaveChangesInterceptor`.

##### UC-DSH-003 — Обновить рецепт

**Тип:** Command. **Статус:** Core. **Этап:** 2.
**Authorization:** POL-001.

Изменение полей Recipe целиком: IntroductionText, ServingsDefault, IsAlcoholic, AuthorTips, ServingSuggestions, Notes. Не затрагивает шаги и ингредиенты — для них отдельные UC (UC-DSH-020..023 и UC-DSH-030..033).

**Правка не трогает `PublishedVersionData`** — аналогично UC-DSH-002.

> **Если автор хочет обновить рецепт целиком (с шагами и ингредиентами):** UI последовательно вызывает несколько UC: UC-DSH-003 + UC-DSH-020..023 + UC-DSH-030..033 + UC-DSH-040..042. Жирный «UpdateRecipeFully» не нужен — это противоречит концепции атомарных UC. На Этапе 4 при реализации UI — паттерн Batched Commit.

##### UC-DSH-004 — Опубликовать блюдо

**Тип:** Command. **Статус:** Core. **Этап:** 2.
**Authorization:** POL-001 (с дополнительным правилом: только автор может **инициировать** публикацию своего Draft; Admin может публиковать любое блюдо).

Двойная семантика:

1. **Первая публикация** (`Status: Draft → Published`):
   - Проверки инвариантов: MainImageId заполнен; Recipe содержит ≥ 1 RecipeStep; Recipe содержит ≥ 1 RecipeIngredient; Timing.TotalTimeMinutes > 0.
   - Собирается полный снепшот агрегата → `PublishedVersionData = <snapshot>`.
   - **Заполняются `*Published`-таблицы** (`DishCategoryPublished`, `DishTagPublished`) из текущих `DishCategory`/`DishTag` — для использования в каталожном фильтре.
   - `PublishedAt = now()`.
   - `PublishedVersionUpdatedAt = now()`.
   - Публикация доменного события `DishPublishedEvent` (Этап 5+).

2. **Повторная публикация** (`Status: Published`, есть несохранённые правки):
   - Те же проверки инвариантов (на случай если правки нарушили целостность).
   - Пересборка снепшота → `PublishedVersionData` обновляется.
   - **Перезапись `*Published`-таблиц** для блюда: старые записи удаляются, новые копируются из текущих `DishCategory`/`DishTag`.
   - `Status` остаётся `Published`. **`PublishedAt = now()`** — фиксирует время последней публикации (для индикатора несохранённых правок).
   - `PublishedVersionUpdatedAt = now()`.
   - Доменное событие `DishUpdatedEvent` (Этап 5+, для подписчиков на автора).

3. **Возврат с Unpublished** (`Status: Unpublished → Published`):
   - Те же проверки инвариантов.
   - Сборка нового снепшота.
   - Заполнение `*Published`-таблиц.
   - `Status = Published`, `PublishedAt = now()`, `PublishedVersionUpdatedAt = now()`.

Все эти операции выполняются **в одной транзакции** — либо всё успешно, либо ничего.

**На Этапе 8+** при `ModerationStatus = Pending` — публикация недоступна автору, ждёт модератора. Сейчас все блюда `Approved` по умолчанию, ограничение неактивно.

##### UC-DSH-005 — Снять блюдо с публикации

**Тип:** Command. **Статус:** Core. **Этап:** 2.
**Authorization:** POL-001.

`Status: Published → Unpublished`. `PublishedVersionData = NULL`. `PublishedAt = NULL`. `PublishedVersionUpdatedAt = NULL`. **Записи в `*Published`-таблицах** (`DishCategoryPublished`, `DishTagPublished`) для этого блюда удаляются. Основные таблицы (`Dish`, `Recipe`, `RecipeStep`, `RecipeIngredient` и пр.) **не затрагиваются** — автор может вернуться к редактированию или повторно опубликовать через `UC-DSH-004`.

`CreatedAt` остаётся неизменным — для UI «когда блюдо появилось» используется именно он.

Все операции — в одной транзакции.

**Связь с другими модулями:**
- Поиск и каталог (UC-DSH-054) перестают показывать блюдо (потому что нет записей в `*Published`).
- Модуль Orders (Этап 6+) хранит собственный снепшот в `OrderItem.DishSnapshot` — участники прошлых заказов видят блюдо как и раньше; новых заказов через каталог сделать нельзя.
- Кэши инвалидируются на Этапе 4+ (когда появятся).

**Опциональное поле причины** (Этап 8+, антиабьюз): автор указывает причину снятия, статистика по причинам — для аналитики и обнаружения паттернов abuse. Сейчас не реализуется.

##### UC-DSH-006 — Архивировать блюдо

**Тип:** Command. **Статус:** Core. **Этап:** 2.
**Authorization:** POL-001.

`Status: any → Archived`. Мягкое удаление — блюдо больше не показывается в каталоге, по прямой ссылке возвращается 404, но данные остаются в БД для возможного восстановления.

**`PublishedVersionData` обнуляется** (если было заполнено). **`PublishedAt = NULL`. `PublishedVersionUpdatedAt = NULL`.**

`CreatedAt` сохраняется — для архивных представлений админа.

**Записи в `*Published`-таблицах** для блюда удаляются (если были) — блюдо исчезает из каталога.

**Связанные медиа НЕ отвязываются** — `IMediaService.DeleteByEntityAsync` НЕ вызывается. Файлы остаются в `EntityType + EntityId`, в статусе `Ready`. Это позволяет:
- Сохранить целостность снепшотов в Orders, которые могут ссылаться на эти медиа.
- Восстановить блюдо в будущем без повторной загрузки фото.
- Hard delete медиа происходит только при hard delete блюда (Этап 8+).

Все операции — в одной транзакции.

**Восстановление из Archived:**
- На Этапе 2 — невозможно для автора.
- На Этапе 8+ — отдельный UC (Drafted, Q-B в открытых вопросах).

**Hard delete блюда** (физическое удаление с каскадом по медиа) — Этап 8+ (Q-C).

**Тип:** Command. **Статус:** Core. **Этап:** 2.
**Authorization:** POL-001.

Это нужно, потому что `SaveChangesInterceptor` не отслеживает изменения связующих таблиц `DishCategory` автоматически — но с точки зрения автора это правка блюда, и индикатор `UpdatedAt > PublishedAt` должен сработать.

#### Чтение блюд (публичный доступ)

##### UC-DSH-050 — Получить публичную карточку блюда по ID

**Тип:** Query. **Статус:** Core. **Этап:** 2.

Возвращает данные блюда без Recipe. Поведение зависит от контекста:

- **Гости / посетители / неавторы:** возвращается **снепшот** из `Dish.PublishedVersionData`. Если снепшота нет (`Status` ∈ {Draft, Unpublished, Archived}) — 404.
- **Автор блюда:** может выбрать через query-параметр `?version=working` (рабочая версия из основных таблиц) или `?version=public` (снепшот, как видят посетители). По умолчанию — `working` для своего блюда, `public` для чужого.
- **Admin:** доступ к обеим версиям.

Параллельно вызывается `UC-DSH-070 IncrementDishViews` (атомарный инкремент счётчика просмотров).

Аналогично UC-DSH-050, но по `slug`. Для SEO-friendly URL: `/dishes/borsch-ukrainskij`. Возвращается **снепшот** (рабочая версия по slug не доступна — slug привязан к публичной версии).

##### UC-DSH-052 — Получить рецепт блюда

**Тип:** Query. **Статус:** Core. **Этап:** 2.

Возвращает Recipe со всеми вложенными сущностями. Источник данных — `Dish.PublishedVersionData` (т.е. рецепт берётся из снепшота, а не из основных таблиц).

**Доступ:**
- Этап 2 (без подписок): доступно всем авторизованным пользователям.
- Этап 3+: добавляется проверка через `ISubscriptionService` — Premium+ или автор блюда. Гости — 401.

Так же как в UC-DSH-050, автор может получить рабочую версию через `?version=working`.

##### UC-DSH-053 — Получить мои черновики

**Тип:** Query. **Статус:** Core. **Этап:** 2.

Список блюд текущего пользователя с фильтрацией по статусу:

- По умолчанию — `Status ∈ {Draft, Unpublished}` (всё, что не опубликовано).
- Опциональный параметр `?withUnsavedChanges=true` — только блюда с `UpdatedAt > PublishedAt` (есть несохранённые правки относительно публичной версии). Полезно для индикатора в личном кабинете «У вас 3 блюда с несохранёнными изменениями».
- Пагинация.

Авторизация — Authenticated.

##### UC-DSH-054 — Поиск и фильтрация блюд

**Тип:** Query. **Статус:** Core. **Этап:** 2.

Параметры: `text`, `categoryIds[]`, `tagIds[]`, `dietLabelsMask`, `difficulty[]`, `cost[]`, `maxTotalTimeMinutes`, `minRating`, `sortBy` (CreatedAt/Rating/Views), пагинация.

**Источник данных:** только блюда с `PublishedVersionData IS NOT NULL` (опубликованы и не сняты) И `ModerationStatus = Approved`. Возвращается **карточка из снепшота**, без рецепта.

**Фильтрация по связям M:M (категории, теги)** идёт через `*Published`-таблицы (`DishCategoryPublished`, `DishTagPublished`). Это критически важно: правки автора в `DishCategory`/`DishTag` не отражаются в каталоге до публикации.

> **Технический момент:** поиск по тексту может идти по основным таблицам (Name, Description) или по `PublishedVersionData` (jsonb-индекс). На Этапе 2 — по `PublishedVersionData` (через `->>` оператор и индекс на jsonb), потому что текст в основных таблицах может не соответствовать опубликованной версии. На Этапе 8+ возможен переход на Elasticsearch / OpenSearch — это будет отдельный ADR.

##### UC-DSH-055 — Получить блюда автора

**Тип:** Query. **Статус:** Core. **Этап:** 2.

Все опубликованные блюда (`PublishedVersionData IS NOT NULL`) указанного `UserId`. Для гостей. Пагинация. Используется на странице профиля автора.

**Связь автора с блюдом — иммутабельна.** Поле `Dish.AuthorUserId` устанавливается при создании и **никогда не обнуляется** — даже при удалении аккаунта автора. Это даёт несколько важных свойств:

- История публикаций сохраняется: каталог по-прежнему может показать «блюда от такого-то автора», даже если аккаунт удалён.
- Сортировка и фильтрация по автору работают одинаково для активных и удалённых аккаунтов.
- Аналитика и статистика по авторам не ломается.

**Удаление аккаунта** (Этап 5+) реализуется в модуле Users как soft delete (`UserProfile.IsDeleted = true`). При выдаче карточки JOIN с `UserProfile` определяет, как показать имя автора:

- `IsDeleted = false` — актуальное `Username` / `DisplayName`.
- `IsDeleted = true` — `DisplayNameAtDeletion` (имя на момент удаления, сохранённое в snapshot) с пометкой «(аккаунт удалён)». См. Q-A.

##### UC-DSH-056 — Пересчитать ингредиенты на N порций

**Тип:** Query. **Статус:** Core. **Этап:** 2.

Возвращает список ингредиентов с количеством, пересчитанным на запрошенное число порций (`servingsRequested / Recipe.ServingsDefault`).

**Доступ:** аналогично UC-DSH-052 (Premium+ на Этапе 3+).

**Источник данных:** `PublishedVersionData` (рецепт из снепшота), при этом сервер использует справочники `Ingredient` и `MeasureUnit` для конвертации единиц (Mass ↔ Volume через `Ingredient.DensityApprox`).

> Сервер выполняет расчёт, а не клиент — чтобы (а) клиент не видел приватной формулы, (б) единицы конвертировались согласованно во всех клиентах (web, mobile, desktop), (в) при изменении формулы пересчёта не требовалось обновлять клиенты.

##### UC-DSH-083 — Откатить мои несохранённые правки блюда

**Тип:** Command. **Статус:** Drafted. **Этап:** 5 или 8+.
**Authorization:** POL-001.

Автор отказывается от текущих несохранённых правок и возвращает рабочую версию к состоянию `PublishedVersionData`. Применимо только к блюдам в статусе `Published` с условием `UpdatedAt > PublishedAt` (есть несохранённые правки).

**Механизм:**
1. Из `Dish.PublishedVersionData` читаются все поля.
2. В одной транзакции переписываются: основные таблицы `Dish`, `Recipe`, `RecipeStep`, `RecipeIngredient`, `DishCategory`, `DishTag`, `Timing`, `Yield`, `Nutrition`.
3. `Dish.UpdatedAt = Dish.PublishedAt` (синхронизация — индикатор «несохранённых правок» становится false).

**Важно:** не путать с откатом до предыдущей версии (Этап 8+, через `DishVersion`). Здесь — только откат к **последней опубликованной** версии.

**Применение:**
- Автор начал править, передумал, хочет вернуть как было.
- Альтернатива авто-сбросу черновиков (от которого мы отказались) — даём явную кнопку под управлением автора.

##### UC-DSH-132 — Объединить теги (admin)

**Тип:** Command. **Статус:** Drafted. **Этап:** 8+.

Слияние дубликатов (например, «без-глютена» и «безглютена» — оставляем второй как канонический, первый удаляем).

**Механизм работы:**

1. В одной БД-транзакции:
   - Для всех записей `DishTag WHERE TagId = <merged_id>` поле `TagId` меняется на `<canonical_id>`.
   - Аналогично для `DishTagPublished` (если запись есть).
   - Дедуплицируются дубли: если у блюда были оба тега, остаётся один.
   - `Tag.UsageCount` пересчитывается у канонического тега.
   - Запись `Tag` с `Id = <merged_id>` удаляется.
   - **В журнал `DishSnapshotInvalidation`** добавляются записи (`DishId`, `Reason = TagsMerged`) для всех затронутых блюд.

2. Фоновая задача `RecalculatePublishedSnapshots` (см. концептуальный раздел) подхватывает записи из журнала и пересобирает `PublishedVersionData` для каждого блюда — чтобы в каталоге отображалось каноническое имя тега.

> **Почему так сложно:** связь Dish ↔ Tag хранится через `DishTag(DishId, TagId)` и `DishTagPublished(DishId, TagId)` — в обеих нет имени тега, только ссылка на ID. После переключения ID — основные данные обновлены автоматически (через JOIN читается актуальное имя). Но снепшот `PublishedVersionData` хранит имена тегов **строками** для быстрой отдачи без JOIN, и его нужно обновить отдельно через журнал invalidation.
