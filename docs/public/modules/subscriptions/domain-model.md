## 1. Ключевые архитектурные решения

| № | Вопрос | Решение |
|---|--------|---------|
| 1 | Структура модуля | Две категории сущностей: **каталог-продукт** (что продаём — декларативная конфигурация) и **биллинг-процесс** (что происходит во времени с подпиской) |
| 2 | Перечень услуг платформы | **Enum `FeatureGrant`**, не таблица-справочник. Каталог услуг стабилен и мал; enum заодно служит code-level реестром реализованных фич |
| 3 | Состав услуг плана | **Join-таблица `PlanGrant`** (план ↔ значение enum). Состав остаётся данными → перекомпоновка планов и промо-подмена без деплоя |
| 4 | Скидки | **Не отдельные планы, а отдельные офферы `PlanPrice`** с собственным окном доступности. Сезонная регулировка процента = активация нужного скидочного оффера, а не правка строки |
| 5 | Переход при успешном продлении | Ссылка `PlanPrice.RenewsAsPriceId` (self-FK). `null` = продлевается как сам себя |
| 6 | Переход при провале списаний | Ссылка `PlanPrice.FallbackPriceId` (self-FK) — dunning-лестница. `null` = терминал (отключить рекуррент) |
| 7 | Гранты при понижении по Fallback | **Сохраняются.** Удешевлённый оффер указывает на тот же план — меняется только цена, доступ не урезается |
| 8 | Расписание ретраев | В **`appsettings`** (число попыток на ступень, интервалы). В подписке хранится только **счётчик** текущих неуспешных попыток |
| 9 | Цена действующей подписки | **Снепшот в `UserSubscription`** (grandfathering). Повышение цены действующим — только на **границе продления** с предварительным уведомлением, не молча и не в середине периода |
| 10 | Договор-оферта | **Снепшот `SubscriptionAgreement`** (jsonb), иммутабельный, append-only. Новая версия — только при материальном изменении условий (цена/план/понижение); обычное продление с неизменными условиями версию не плодит. Паттерн повторяет jsonb-снепшот Dishes |
| 11 | Промоакции | **Оверлей грантов** (`Promotion` + `PromotionTarget` + `PromotionGrant`). Промо меняет только гранты, не цены; базовый план не трогает |
| 12 | Покупочный роль-гейт | Поле `RequiredRole` на плане — **порог** «не ниже роли» (высокая роль может купить низкий план). Проверка eligibility только при **покупке/апгрейде**, не на продлении. KYC делегируется Users через `IRoleEligibilityService`. У `AddOn`-планов роль-гейта нет |
| 13 | Идемпотентность webhook | UNIQUE на `SubscriptionPayment.GatewayTransactionId`; повтор доставки = no-op |
| 14 | Безопасность webhook | Эндпоинт не авторизуется JWT (дёргает шлюз). Верификация по IP-диапазонам ЮKassa + перезапрос реального статуса (`GET /payments/{id}`); опционально signing secret. Тело уведомления не считается доверенным |
| 15 | Платёжный шлюз | ЮKassa за абстракцией `IPaymentGateway` (Application объявляет, Infrastructure реализует). Рекуррент инициирует суточный сборщик, webhook завершает переход. См. ADR-0017 `recurring-payments-yookassa` |
| 16 | Корень агрегата | Единственный `AggregateRoot<Guid>` — `UserSubscription`. Остальные — `Entity<Guid>` / join |
| 17 | Резолвер доступа | `ISubscriptionAccessService` — единый источник истины эффективных грантов: **объединение (union)** грантов всех активных подписок (Base + AddOn) `±` промо, с гейтом усвоения и квотами. Аналог `IDishOwnershipPolicy` из POL-001 |
| 18 | Время | `DateTimeOffset` везде; PostgreSQL `timestamp with time zone` |
| 19 | Схема БД | `subscriptions`; отдельный `SubscriptionsDbContext` (при `dotnet ef` указывать `--context`) |
| 20 | Антифрод | По правилам шлюза, своё не строим. Обработка `Failed` ветвится по `cancellation_details.reason`: бизнес-отказ → dunning; фрод-причина → отключение рекуррента (`RecurringDisabledReason = Antifraud`) |
| 21 | Юридическое согласие (сценарий B) | Термы — в `SubscriptionAgreement` + якорь акта (`AcceptedAt`); полный контекст акта (IP/UA/метод/3DS) — событием в бизнес-лог. `ConsentLog` не заводим. Техническая привязка (1 ₽/ноль) ≠ согласие |
| 22 | Мульти-слот Base + AddOn | `PlanKind` (`Base`/`AddOn`). У пользователя **≤1 активной Base** + сколько угодно `AddOn`. Несколько строк `UserSubscription` на пользователя. Роль несёт только Base |
| 23 | Гейт усвоения гранта | Помимо покупочного гейта — реестр `грант → RequiredRole?` (статический в Domain). Роль-привязанный грант (продвижение/реклама → `Chef`) **инертен** без роли; потребительские гранты — агностичны. Это и есть карательное понижение |
| 24 | Карательное понижение роли | Понижение роли (за нарушения) подписку **не отменяет** и возврат **не делает**. Роль-привязанные гранты гаснут через гейт усвоения; потребительские и сама подписка работают; продление не блокируется |
| 25 | Грантовый floor (период привлечения) | Базовый уровень временно открыт всем: при отсутствии активной Base резолвер возвращает floor (= гранты базового плана), управляется флагом конфигурации. Без строк-подписок на каждого пользователя |
| 26 | Квота права | Квотовые гранты (`PromotionAdvanced`) несут число: `PlanGrant.Quantity` (`int?`, `null` = безлимит). Резолвер **суммирует** квоты по активным подпискам. Учёт *использования* — в Dishes, не в Subscriptions |
| 27 | Инварианты лестницы офферов | `RenewsAsPriceId`/`FallbackPriceId` → оффер **того же плана**, цепочки **без циклов**, у `IsRecurring = false` переходов нет. Проверяется валидацией в Application (FK-сравнение CHECK-ом не выразить) |

## 2. Стратегия реализации

Реализация модуля идёт слоями каталог → подписка → удержание, чтобы каждый шаг давал работающий инкремент:

1. **Каталог-продукт:** `SubscriptionPlan`, `PlanGrant`, `PlanPrice` + enum `FeatureGrant`, `OfferKind`. Seed базовых планов (Free / Premium / Business) и их офферов.
2. **Подписка и платежи:** агрегат `UserSubscription`, `SubscriptionPayment`, резолвер `ISubscriptionAccessService`. Mock платёжного шлюза для разработки.
3. **Оферта:** `SubscriptionAgreement` (снепшот условий), генерация первичной версии при оформлении.
4. **Промоакции:** `Promotion`, `PromotionTarget`, `PromotionGrant` + наложение в резолвере.
5. **Рекуррентные списания и webhook:** интеграция с ЮKassa/CloudPayments, обработчик webhook, фоновое продление (архитектура сервиса списаний — см. отдельное обсуждение, [[раздел 13]]).

Repository-реализации создаются по мере появления первого UC-потребителя (соглашение проекта). Гранты на грантах (`SaveChangesAsync` на каждом repository-интерфейсе) — для единообразия.

## 4. Концептуальные основы

### 4.1. Разделение «каталог-продукт» и «биллинг-процесс»

Главное решение модуля — не смешивать *что мы продаём* с *тем, что происходит во времени*:

- **Каталог-продукт** (`SubscriptionPlan`, `PlanGrant`, `PlanPrice`) — декларативная конфигурация. Стабилен, редактируется редко (маркетинг/продакт).
- **Биллинг-процесс** (`UserSubscription`, `SubscriptionPayment`, переходы) — поведение во времени: списания, ретраи, понижение при неудаче, продление.

Из-за смешения этих категорий ранние черновики пытались выразить процесс (что делать при рекуррентном платеже) через структуру каталога (ссылки тариф→тариф). Это плодило дубли планов. В финальной модели «следующий оффер» — это свойство **цены** (`PlanPrice`), а счётчики и состояние — свойство **подписки** (`UserSubscription`).

### 4.2. Гранты через enum и резолвинг эффективного доступа

Услуги платформы — это enum `FeatureGrant` (раздел 5). Преимущества: каталог реализованных услуг виден прямо в коде; добавление услуги — осознанный акт (новое значение enum) при выкатке глобальной фичи. Состав плана из грантов хранится в join-таблице `PlanGrant`, то есть остаётся данными.

**Два независимых гейта по роли.** Их нельзя путать:

- **Покупочный** (`SubscriptionPlan.RequiredRole`) — кто может *купить* Base-план (порог «не ниже роли»). Проверяется при покупке/апгрейде.
- **Усвоения** (реестр `грант → RequiredRole?` в Domain) — работает ли грант *в рантайме* без роли. Большинство грантов агностичны (`null`); роль-привязанные (продвижение/реклама → `Chef`) инертны без роли.

**Эффективный набор грантов** собирает `ISubscriptionAccessService` как **объединение по всем активным подпискам** пользователя (Base + AddOn):

```
эффективные гранты =
    ∪ грантов всех активных подписок (Base + AddOn) через PlanGrant
  ∪ гранты добавляющих промо (PromotionGrant, IsGrant = true), чьи таргеты матчат пользователя
  − гранты убирающих промо  (PromotionGrant, IsGrant = false)
  затем фильтр гейта усвоения: грант остаётся, если у него нет RequiredRole ИЛИ роль у пользователя есть
```

- Промо фильтруются по сроку (`Promotion.From..To`) и активности.
- **Квота:** для квотовых грантов (`PromotionAdvanced`) резолвер возвращает не флаг, а **сумму** `PlanGrant.Quantity` по всем активным подпискам (`null` = безлимит). Учёт расхода — в Dishes.
- **Грантовый floor (период привлечения):** при отсутствии активной Base резолвер подмешивает базовый набор (= гранты базового плана), включаемый флагом конфигурации. Так базовый уровень временно открыт всем зарегистрированным без строк-подписок.

Резолвер — единая точка проверки; контроллеры и хендлеры других модулей зависят только от него (аналог `IDishOwnershipPolicy`).

**Карательное понижение роли.** Понижение роли применяется только за нарушения. Подписка при этом **не отменяется** и возврат **не делается**: роль-привязанные гранты просто гаснут через гейт усвоения, а потребительские гранты и сама подписка (включая продление) продолжают работать. Отдельной обработки в машине состояний это не требует — всё решается в резолвере.

**Правило при понижении по Fallback:** гранты сохраняются. Удешевлённый оффер (`DunningFallback`) ссылается на тот же `SubscriptionPlan`, поэтому при переходе меняется цена, но набор `PlanGrant` остаётся прежним.

### 4.3. Две ссылки «следующий оффер»: `RenewsAs` (успех) и `Fallback` (провал)

У оффера два разных «следующих оффера», и их нельзя путать:

| Ссылка | Когда срабатывает | Смысл |
|--------|-------------------|-------|
| `RenewsAsPriceId` | На **успешном** продлении периода | Перейти на этот оффер при следующем списании. `null` = продлевается как сам себя |
| `FallbackPriceId` | При **провале** списаний после исчерпания ретраев | Понизить на этот оффер (dunning-лестница). `null` = терминал, отключить рекуррент |

Примеры: триал 3д и интро 7д имеют `RenewsAs` → 30д базовый (продление в оффер другой длительности); 30д/90д/365д базовые имеют `RenewsAs = null` (продлеваются собой) и `Fallback` → удешевлённый.

### 4.4. Версионирование цен, grandfathering, repricing на продлении

`UserSubscription` хранит **снепшот цены** (`SnapshotAmount`, `SnapshotCurrency`) — клиент платит зафиксированную сумму, даже если цена оффера в каталоге выросла. Это grandfathering по умолчанию, и поэтому `UserSubscription → PlanPrice` обязательно `OnDelete: Restrict` (цену, по которой кто-то живёт, нельзя удалить).

Чтобы **поднять цену действующим** подпискам:

1. Завести новую версию `PlanPrice` с новым `Amount`. Новые покупатели получают её автоматически.
2. Текущий период действующих подписок не трогать. На **границе следующего продления** перевести подписку на актуальную цену.
3. Перед списанием по новой цене — предварительное уведомление (Notifications, Этап 5) с правом отказаться (`CancelAtPeriodEnd = true`).

Повышение — всегда явное действие на границе периода, никогда не тихая правка снепшота.

### 4.5. Оферта-снепшот (договор)

При оформлении подписки фиксируется неизменяемый слепок условий — `SubscriptionAgreement` (jsonb-снепшот: название плана, гранты, длительность, сумма, валюта, скидка, период). Это юридический след «на что именно соглашался клиент».

Таблица **иммутабельна и append-only**: новая версия (`Version` + `ChangeType`) пишется **только при материальном изменении условий** — цена (`PriceChange`), план (`PlanChange`), понижение по Fallback (`Downgrade`). Обычное автопродление с неизменной ценой и базовой информацией версию **не создаёт**: факт продления виден в `SubscriptionPayment` и полях периода `UserSubscription`. Так таблица не разрастается на каждом цикле списания. Старые версии остаются как доказательство прежних условий. Паттерн повторяет двухслойную модель Dishes (`PublishedVersionData jsonb`): денормализованный неизменяемый слепок для аудита, отдельно от операционных полей.

`SnapshotAmount` в `UserSubscription` и снепшот в `SubscriptionAgreement` — не дубль: первое операционное (для следующего списания), второе юридическое (полный слепок условий каждой версии).

### 4.6. Промоакции как оверлей грантов

Промо добавляет или убирает **гранты** (не цены) на срок акции, не трогая базовый план. Таргетинг — по роли, плану или конкретному пользователю (`PromotionTarget`). «Автоматом выдать промо держателям тарифа X» = строка `PromotionTarget(TargetType = Plan, TargetValue = X)`. Скидки промоакциями не выражаются — это офферы (`PlanPrice`).

### 4.7. Покупочный роль-гейт и делегирование KYC в Users

Поле `SubscriptionPlan.RequiredRole` ограничивает **покупку** Base-плана и работает как **порог** «не ниже роли»: повар может купить и потребительский тариф (у потребительских планов `RequiredRole = null`). Use-case `Subscribe`/апгрейд перед созданием подписки вызывает `IRoleEligibilityService.IsEligibleAsync(userId, requiredRole)`. На **продлении** eligibility **не перепроверяется** (добровольного понижения роли нет, карательное подписку не гасит). У `AddOn`-планов покупочного роль-гейта нет — их докупает любой.

Это покупочный гейт; рантайм-доступ к гранту регулирует отдельный **гейт усвоения** (см. 4.2).

Сам процесс перехода в роль Chef/Restaurant (проверка ИНН, банковских реквизитов, договора с площадкой, документов об образовании) — **не Subscriptions**, а Users / verification-флоу (Этап 6, где `BusinessProfile` с ИНН). Subscriptions только потребляет статус eligibility. На Этапе 3 `IRoleEligibilityService` может быть заглушкой. Паттерн повторяет `IAuthUserService` (Вариант 2): слабая зависимость через интерфейс.

### 4.8. Рекуррентные платежи: расписание ретраев в `appsettings`

*Число попыток* и *интервалы* dunning — это процесс-политика, она в конфигурации, а не в БД. В подписке хранится только счётчик `FailedAttempts` (состояние конкретной подписки).

```jsonc
"Subscriptions": {
  "Dunning": {
    "MaxAttemptsPerStage": 3,          // попыток на одной ступени лестницы
    "RetryIntervalsHours": [24, 72, 168] // паузы между попытками: +1д, +3д, +7д
  }
}
```

Логика: при провале списания `FailedAttempts++`; если `FailedAttempts >= MaxAttemptsPerStage` — переход на `FallbackPriceId` (если не `null`), сброс счётчика, попытка списать по новой цене; если `Fallback = null` — `AutoRenew = false`, статус `Expired`. Полная машина состояний — отдельный документ (раздел 13).

### 4.9. Мульти-слот: Base + AddOn

`SubscriptionPlan.PlanKind` различает два рода планов:

- **`Base`** — тарифный уровень (Любитель, Большая семья, Базовый повара, Про повар…). У пользователя **ровно одна активная Base**-подписка. Несёт роль (эмитит события смены роли). Имеет покупочный роль-гейт.
- **`AddOn`** — докупаемая услуга «à la carte» (например, «+10 продвижений», «доп. реклама»), живёт **параллельно** Base. Активных AddOn — сколько угодно (но не дубль одного и того же одновременно). Роль не несёт, покупочного роль-гейта не имеет.

Технически мульти-слот — это просто несколько строк `UserSubscription` у пользователя; каждая со своим периодом, офертой, расписанием и dunning. Резолвер объединяет их гранты (см. 4.2). Сборщик списаний и машина состояний работают с каждой подпиской одинаково; события смены роли эмитит только Base.

**Скоуп Этапа 3–4:** реализуем только Base-тарифы (вплоть до повара), без AddOn и без proration. Но структуру (`PlanKind`, мульти-слот, резолвер-union) закладываем сразу — иначе добавление AddOn позже стало бы ломающей миграцией. Сейчас union тривиален (одна Base + ноль AddOn).

Записи только добавляются, никогда не правятся. UNIQUE `(SubscriptionId, Version)`.

**Согласие (сценарий B):** оферта хранит *что* принято (`TermsSnapshot`, `ContentHash`) + якорь акта (`AcceptedAt`, `Version`). *Полный контекст акта* (IP, User-Agent, метод подтверждения, 3-D Secure) пишется не в домен, а в **бизнес-лог** через событие `ConsentGiven`/`SubscriptionCreated` при оформлении. Отдельный `ConsentLog` не заводится — версионирование оферты служит журналом согласий, а акт как юридически значимое действие — территория бизнес-лога. Механизм отписки обязателен (ЮKassa проверяет его при подключении рекуррента).

### 6.7. `Promotion` — промоакция (`Entity<Guid>`)

## 7. Раскладка градации тарифов в офферы (worked example)

План с двумя грантами (`FullRecipes`, `IngredientCalculator`) раскладывается в строки `PlanPrice` так (цифры условные):

| Оффер | Kind | DurationDays | Amount | IsPurchasable | RenewsAs → | Fallback → |
|-------|------|:------------:|:------:|:-------------:|-----------|-----------|
| Триал 3д | `Trial` | — (TrialDays=3) | 0 | да | 30д база | — |
| 7д скидка | `Intro` | 7 | 199 | да | 30д база | — |
| 30д скидка | `Intro` | 30 | 449 | да | 30д база | — |
| 30д база | `Standard` | 30 | 599 | да | `null` (сам) | Удешевлённый 30д |
| 90д скидка | `Intro` | 90 | 1290 | да | 90д база | — |
| 90д база | `Standard` | 90 | 1690 | да | `null` (сам) | Удешевлённый 30д |
| 365д скидка | `Intro` | 365 | 4990 | да | 365д база | — |
| 365д база | `Standard` | 365 | 6900 | да | `null` (сам) | 30д база |
| Удешевлённый 30д | `DunningFallback` | 30 | 299 | **нет** | `null` (сам) | `null` (стоп) |
| Всё включено (retention) | `Retention` | 30 | 399 | **нет** | себя или удорожание | Удешевлённый 30д |

Триал/7д/30д-скидка ведут в «30д база» через `RenewsAs`. Dunning-лестница идёт по `Fallback`: базовый → удешевлённый → стоп. Гранты на всех ступенях одного плана одинаковы (см. 4.2).

## 8. OnDelete-стратегии

Стратегии выбраны осознанно (как в Dishes):

| FK | Стратегия | Обоснование |
|----|-----------|-------------|
| `PlanGrant.PlanId → SubscriptionPlan` | `Cascade` | Связка состава умирает вместе с планом |
| `PlanPrice.PlanId → SubscriptionPlan` | `Restrict` | План с офферами не удаляют, гасят через `IsActive` |
| `PlanPrice.RenewsAsPriceId → PlanPrice` | `Restrict` | Нельзя удалить оффер, на который ссылаются как на цель продления |
| `PlanPrice.FallbackPriceId → PlanPrice` | `Restrict` | Нельзя удалить оффер — цель понижения |
| `UserSubscription.CurrentPriceId → PlanPrice` | `Restrict` | Защита grandfathering: цену, по которой кто-то живёт, удалять нельзя |
| `UserSubscription.PlanId → SubscriptionPlan` | `Restrict` | Защита от потери ссылки на продукт |
| `SubscriptionPayment.SubscriptionId → UserSubscription` | `Cascade` | Журнал платежей принадлежит подписке |
| `SubscriptionPayment.PriceId → PlanPrice` | `Restrict` | Сохранить ссылку на оффер для аудита |
| `SubscriptionAgreement.SubscriptionId → UserSubscription` | `Cascade` | Оферта принадлежит подписке |
| `PromotionTarget.PromotionId → Promotion` | `Cascade` | Таргеты принадлежат акции |
| `PromotionGrant.PromotionId → Promotion` | `Cascade` | Гранты акции принадлежат акции |

## 9. CHECK-constraints

Defense in depth поверх FluentValidation:

| Таблица | Ограничение |
|---------|-------------|
| `PlanPrice` | `Amount >= 0` |
| `PlanPrice` | `Kind = Trial` ⇒ `TrialDays IS NOT NULL AND Amount = 0` |
| `PlanPrice` | `AvailableFrom < AvailableUntil` (когда оба заданы) |
| `UserSubscription` | `CurrentPeriodStart < CurrentPeriodEnd` |
| `UserSubscription` | `FailedAttempts >= 0` |
| `SubscriptionPayment` | `Amount >= 0` |
| `SubscriptionPayment` | UNIQUE(`GatewayTransactionId`) — идемпотентность webhook |
| `Promotion` | `From < To` |
| `SubscriptionAgreement` | UNIQUE(`SubscriptionId`, `Version`) |
| `PlanPrice` (app-валидация) | `RenewsAsPriceId`/`FallbackPriceId` → тот же `PlanId`; цепочки без циклов; у `IsRecurring = false` переходов нет |
| `UserSubscription` (app-валидация + частичный UNIQUE) | ≤1 активной Base-подписки на пользователя |

Последние две — на уровне Application (FK-сравнение и обход цепочки CHECK-ом не выразить), частичный UNIQUE-индекс по Base в активных статусах — как defense-in-depth.

Публикуются после коммита транзакции через RabbitMQ (Integration Events, пересекают границу модулей):

## 11. Авторизация

Правила «кто может оформить/отменить/изменить подписку» выносятся в политику **`POL-004 Subscription Access`** (по образцу POL-001). UC ссылаются на политику через `Authorization: POL-004`, а не дублируют правила. Eligibility роль-гейта (раздел 4.7) — часть этой политики.

Проверка эффективных грантов в защищённых эндпоинтах других модулей идёт через `ISubscriptionAccessService.HasFeatureAsync(userId, FeatureGrant.X)`.

## 12. Отложенное (Deferred)

| Что | Когда |
|-----|-------|
| **AddOn-планы** (`PlanKind = AddOn`) — докупаемые услуги параллельно Base | Этап 4+ (структура заложена сейчас) |

| **Proration** — пропорциональный зачёт остатка при смене среди периода | Этап 4+ |

| **Free-Base план** — постоянный бесплатный уровень (сейчас базовый временно открыт floor-флагом) | По ситуации |

| Несколько валют | По мере выхода на мультивалютность |

## 13. Открытые вопросы и что дальше

**Решено в этой версии:** гранты сохраняются при понижении по `Fallback`; расписание ретраев — в `appsettings`; версия оферты создаётся только при материальном изменении условий (не на каждом продлении).

**Следующие шаги обсуждения:**

1. **Машина состояний `UserSubscription`** — переходы `Scheduled → Trialing → Active → PastDue → (Fallback) → … → Canceled/Expired`, момент инкремента `FailedAttempts`, обработка идемпотентного webhook, точные точки публикации `SubscriptionActivated`/`Expired`. Зафиксирована в `subscription-state-machine.md`.
2. **Архитектура рекуррентных списаний** — единый `PaymentService` против отдельного сервиса рекуррентов; режим работы фонового сборщика (раз в час / раз в день / однократный запуск) против приёма webhook от ЮKassa в `PaymentService`. Зафиксирована в `../../adr/ADR-0017-recurring-payments-yookassa.md`.
