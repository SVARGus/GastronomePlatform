## Легенда Phase (фазирование внутри Этапа 3)

Этап 3 разбит на три инкрементальные фазы. Колонка `Phase` в сводной таблице ниже фиксирует, к какой фазе относится каждый UC:

| Phase | Значение |
|-------|----------|
| **A** | Phase A — Catalog + Subscribe happy-path + Access. Mock `IPaymentGateway`, синхронные транзакции, скелет планировщика. State machine: `Trialing`/`Active`/`Canceled`/`Expired`. Без промо, retention, dunning fallback, AddOn. |
| **B** | Phase B — реальный биллинг через ЮKassa: `YooKassaPaymentGateway`, асинхронный webhook, reconciliation подвисших платежей, антифрод по правилам шлюза, понижение по Fallback и dunning-ретраи (`PastDue`). **Вынесена за пределы текущего объёма работ** — см. ниже. |
| **C** | Phase C — полировка и расширения: промо-оверлей, retention-флоу, отложенные Scheduled-подписки, реактивация, смена способа оплаты, редактирование каталога. |
| **—** | За пределами Этапа 3 — Drafted/Deferred на других этапах (4+/5/6/8+) или Deferred без этапа. |

### Статус Phase B

Все сценарии Phase B требуют подключения к боевому магазину ЮKassa: они реагируют на реальные уведомления шлюза, дочитывают статусы платежей и ветвятся по причинам отказа. Подключение упирается во внешнюю зависимость с непредсказуемым сроком (регистрация магазина, боевые ключи, включение автоплатежей по запросу менеджеру), поэтому боевой адаптер не реализуется, а UC этой фазы переведены в `Drafted` с этапом 4+.

Это не пробел, а спроектированная граница: провайдер скрыт за портом `IPaymentGateway`, реализован адаптер `MockPaymentGateway`, и появление боевого адаптера потребует изменений только в Infrastructure. Целевая архитектура интеграции зафиксирована в `ADR-0017` (раздел «Границы реализации» перечисляет, что именно реализовано), механика переходов — в `subscription-state-machine.md`.

## Концепция: каталог-продукт vs биллинг-процесс

Модуль делится на две категории (подробно — в `../domain-model.md`):

- **Каталог-продукт** — что продаём: `SubscriptionPlan`, `PlanGrant`, `PlanPrice`. Декларативная конфигурация, управляется админом (UC-SUB-001..010).
- **Биллинг-процесс** — что происходит во времени: `UserSubscription`, списания, ретраи, понижение, продление (UC-SUB-020..025, 200..208).

---

## Концепция: мульти-слот Base + AddOn

`SubscriptionPlan.PlanKind` различает **Base** (тарифный уровень, ≤1 активной, несёт роль) и **AddOn** (докупаемая услуга параллельно Base, роль-агностична). У пользователя несколько строк `UserSubscription`. На Этапе 3–4 реализуем только Base (до повара), без AddOn/proration, но структуру закладываем сразу. Гранты бывают булевыми (1–4), уровневыми и квотовыми (`PromotionAdvanced` — число блюд через `PlanGrant.Quantity`); продвижение/реклама (5–8) — роль-привязаны к `Chef` и инертны до Этапа 4+.

## Концепция: авторизация (POL-004)

Операции над подпиской и доступ к услугам подчиняются единой политике **POL-004 Subscription Access** (см. `../../../policies/POL-004-subscription-access.md`).

**Кратко (Этап 3):**

- Операции над подпиской — только Owner (`UserSubscription.UserId == ActorUserId`) или Admin.
- Оформление Base-плана — покупочный роль-гейт (`RequiredRole` как порог) через `IRoleEligibilityService`; AddOn без гейта.
- Доступ к услуге из других модулей — через `ISubscriptionAccessService.HasFeatureAsync`: эффективные гранты = **union** всех активных подписок (Base + AddOn) ± промо, с **гейтом усвоения** (роль-привязанные гранты инертны без роли) и суммированием квот.
- **Карательное понижение роли** подписку не отменяет и не возвращает деньги — роль-привязанные гранты гаснут через гейт усвоения.

В UC-описаниях вместо явных правил используется ссылка `Authorization: POL-004`.

## Концепция: согласие и бизнес-лог (сценарий B)

Техническая привязка карты (1 ₽ или ноль) **≠** юридическое согласие на рекуррент. Согласие образуют оферта с раскрытыми условиями + явный акт согласия + 3-D Secure + механизм отписки.

- **Что принято** (термы) — снепшот `SubscriptionAgreement` + якорь акта `AcceptedAt`.
- **Контекст акта** (IP, User-Agent, метод) — событием `ConsentGiven`/`SubscriptionCreated` в **бизнес-лог** (сквозной модуль, TODO).
- Отдельный `ConsentLog` не заводится — версионирование оферты служит журналом согласий.

UC оформления/продления/отмены эмитят события в бизнес-лог.

## Концепция: рекуррентные списания (ADR-0017)

Архитектура зафиксирована в `../../../adr/ADR-0017-recurring-payments-yookassa.md`:

- ЮKassa не держит расписание — списания инициирует **суточный сборщик** (UC-SUB-200).
- **PaymentService (webhook)** принимает ответ ЮKassa и двигает машину состояний (UC-SUB-201).
- Шлюз скрыт за `IPaymentGateway`. Идемпотентность на двух концах, reconciliation для подвисших платежей.

Из перечисленного реализованы порт `IPaymentGateway` и адаптер `MockPaymentGateway`; боевой адаптер и webhook-контур вынесены за пределы текущего объёма работ (см. «Статус Phase B» и ADR-0017 §«Границы реализации»).

**Тип:** Qry. **Статус:** Core. **Этап:** 3. **Phase:** A. План ± промо, гейт по текущей роли.

##### UC-SUB-052 — Опубликовать событие активации/истечения
**Тип:** Sys. **Статус:** Core. **Этап:** 3. **Phase:** A. `SubscriptionActivated` → Users: повышение роли при активации Base-подписки. `SubscriptionExpired` → Notifications: уведомление об окончании.

Роль при истечении **не отзывается** — зависимость «роль ↔ подписка» односторонняя (POL-004 §4.4b): роль является предусловием покупки, а доступ к платным возможностям определяют гранты, которые отсекаются по `CurrentPeriodEnd` независимо от статуса подписки и набора ролей.

События эмитируются для планов обоих родов; отбор `Base` выполняет подписчик (см. `../subscription-state-machine.md` §8.2).

## 200–208. Системные сценарии (биллинг)

##### UC-SUB-200 — Инициировать рекуррентные списания
**Тип:** Job. **Статус:** Core. **Этап:** 3. **Phase:** A. Суточный сборщик: выбрать подписки по `NextBillingAt` и `TrialEnd`, создать автоплатежи через `IPaymentGateway` с `Idempotence-Key`, записать `Pending`/`Succeeded`. Статус не двигает в реальном сценарии (двигает webhook); в Phase A с mock-шлюзом — переход применяется синхронно. Phase A — скелет `IHostedService` с таймером; Phase B — продакшен-логика с ЮKassa.

##### UC-SUB-201 — Обработать webhook платежа
**Тип:** EvtH. **Статус:** Drafted. **Этап:** 4+. **Phase:** B. Верификация (IP + перезапрос статуса), идемпотентность по `GatewayTransactionId`, применение перехода машины состояний, публикация событий. Ветвление `Failed` по `cancellation_details.reason` (бизнес-отказ → dunning, фрод → антифрод). Требует боевого шлюза: обрабатывает реальные уведомления ЮKassa.

##### UC-SUB-202 — Понизить подписку по Fallback (dunning)
**Тип:** Sys. **Статус:** Drafted. **Этап:** 4+. **Phase:** B. При исчерпании ретраев: `CurrentPriceId ← FallbackPriceId`, сброс счётчика, оферта `Downgrade`, `SubscriptionDowngraded`. Гранты сохраняются. Запускается результатом webhook (UC-SUB-201), поэтому реализуется вместе с ним.

##### UC-SUB-203 — Истечь подписку по окончании периода
**Тип:** Sys. **Статус:** Core. **Этап:** 3. **Phase:** A. На `CurrentPeriodEnd` без продления: `Expired`, `EndedAt`, `SubscriptionExpired`.

Истекает из `Trialing`, `Active` и `Canceled` — пока автопродление не подключено, неотменённая подписка иначе осталась бы в `Active` навсегда. `PastDue` в охват не входит: этот переход относится к контуру dunning (UC-SUB-202, Phase B). Подробности и побочные эффекты — `../subscription-state-machine.md` §6.

На отсечение доступа сборщик не влияет: гранты перестают резолвиться по `CurrentPeriodEnd` независимо от статуса, поэтому лаг сборщика сказывается только на актуальности поля `Status`.

##### UC-SUB-204 — Reconciliation подвисших `Pending`
**Тип:** Job. **Статус:** Drafted. **Этап:** 4+. **Phase:** B. Для платежей без пришедшего webhook — дочитать статус через `GET /payments/{id}` и синхронизировать. Требует боевого шлюза: страхует именно ненадёжную доставку уведомлений, которой у mock-адаптера нет.

##### UC-SUB-205 — Вернуть проверочное списание (1 ₽)
**Тип:** Sys. **Статус:** Core. **Этап:** 3. **Phase:** A. Синхронный refund в обработчике успешного verification: в Phase A пишем `SubscriptionPayment(Refunded)` рядом с `(Verification, Succeeded)` в одной транзакции через mock-шлюз; в Phase B — реальный `POST /refunds` плюс idempotent webhook `refund.succeeded`. Только путь 1 ₽.

##### UC-SUB-206 — Отключить рекуррент по антифроду
**Тип:** Sys. **Статус:** Drafted. **Этап:** 4+. **Phase:** B. По фрод-причине шлюза: `AutoRenew = false`, `RecurringDisabledReason = Antifraud`, без ретраев. Требует боевого шлюза: решение принимается по значению `cancellation_details.reason` из реального ответа ЮKassa.

##### UC-SUB-207 — Активировать отложенную (Scheduled) подписку
**Тип:** Sys. **Статус:** Core. **Этап:** 3. **Phase:** C. На `StartsAt`: первое списание, переход `Scheduled → Active`, `SubscriptionActivated`. Используется только retention-флоу (UC-SUB-081), который сам Phase C.

##### UC-SUB-208 — Обработать `RoleChanged` из Users
**Тип:** EvtH. **Статус:** Drafted. **Этап:** 4. **Phase:** —. **Карательное понижение** (только за нарушения): подписку **не отменяет**, возврат не делает, продление не блокирует. Роль-привязанные гранты гаснут автоматически через гейт усвоения в резолвере; обработчик лишь логирует/уведомляет.
