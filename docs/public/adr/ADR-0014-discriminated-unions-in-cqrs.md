## 1. Context (Контекст)

В предметной области проекта встречаются и будут встречаться **дискриминированные объединения** — сущности с двумя или более «природами», каждая со своим набором полей, объединённые в одной таблице через дискриминирующий признак. C# 12 / .NET 8 не поддерживают discriminated unions (DU, sum types в терминологии теории типов) как нативную конструкцию языка — соответствующий proposal `Type Unions` находится в активном обсуждении `dotnet/csharplang`, но в текущей версии языка отсутствует. Это значит каждое появление DU-сущности в проекте требует осознанного выбора паттерна.

Первый такой кейс — `RecipeIngredient` (catalog-позиция из справочника `Ingredient` vs freeform-позиция со свободным текстом). Из дорожной карты видны дальнейшие применения — см. §6.

Без зафиксированного общего принципа риск — на каждом следующем DU-кейсе заново разбирать те же альтернативы и принимать разные решения. Этот ADR фиксирует принцип один раз и используется как reference во всех будущих узких ADR по конкретным DU-сущностям.

Контекст принятия решения дополнительно определяется существующей CQRS-моделью проекта (ADR-0003 — зарезервирован): операции разделены на `ICommand` / `IQuery` через MediatR, в каждом UC присутствует отдельный DTO на стороне Read. Запросы и команды работают через один и тот же `DbContext`, источник данных общий — это «CQRS lite», без отдельных read-моделей и event sourcing. Read-стороны и Write-стороны уже **разделены по типам** (`Dish` ≠ `DishCardDto` ≠ `DishDetailDto`); вопрос лишь в том, делать ли это разделение **разнотипным** для одной DU-сущности.

## 2. Considered Alternatives (Рассмотренные альтернативы)

### Вариант A — Один класс с nullable-полями + runtime XOR

Write-side: один Domain-класс с публичным конструктором/фабрикой, принимающим все возможные поля как nullable. XOR-инвариант проверяется в runtime внутри конструктора. Read-side: один DTO с тем же набором nullable-полей.

- **Плюсы:** минимум кода; вся логика в одном месте.
- **Минусы:** компилятор не помогает — невалидные комбинации параметров обнаруживаются только в runtime (или в тестах); потребители read-DTO видят N nullable-полей и обязаны мысленно держать XOR-инвариант в голове («если поле X не null — поле Y гарантированно null»). Любая ошибка читателя — `NullReferenceException` или подмена данных.
- **Отклонён**: жертвует системой типов в обе стороны.

### Вариант B — Множество фабрик на одном классе, один общий read-DTO

Write-side: одна Domain-сущность с приватным конструктором и фабриками по природе (`CreateFromCatalog`, `CreateFreeform`). Read-side: единый DTO с nullable-полями (как в Варианте A).

- **Плюсы:** write-side получает структурную гарантию инварианта на этапе компиляции.
- **Минусы:** read-side остаётся уязвимым к ошибкам потребителя; nullable-поля DTO накладывают когнитивную нагрузку на UI/клиента; рост числа специфичных полей у одной природы (`IngredientImageUrl` только для catalog) дальше раздувает общий DTO.
- **Отклонён**: половинчатое решение — защищает только запись.

### Вариант C — Иерархия классов (TPH в EF Core)

Базовый Domain-класс + конкретные наследники (`CatalogRecipeIngredient`, `FreeformRecipeIngredient`). EF Core маппит как TPH (Table Per Hierarchy) с дискриминирующей колонкой.

- **Плюсы:** идиоматично для DDD, типобезопасно с обеих сторон.
- **Минусы:** TPH-маппинг разрезает запросы по типу (фильтры/`Where` по конкретному типу — отдельная EF-механика); миграции становятся сложнее; ломает принцип «один тип — одна таблица», что усложняет ментальную модель схемы; добавление новой природы требует не просто нового наследника, но и пересмотра дискриминатора в EF-конфигурации.
- **Отклонён**: цена сложности EF-маппинга превышает выигрыш по сравнению с (E) для типичных DU-кейсов в проекте.

### Вариант D — Value Object discriminator (`IngredientSource`)

Внутри `RecipeIngredient` — value object `IngredientSource` с фабриками `FromCatalog(...)` / `Freeform(...)`. Сама сущность принимает `IngredientSource` в конструкторе.

- **Плюсы:** инкапсулирует дискриминацию в отдельный тип; алгебраическая чистота.
- **Минусы:** value object для одного поля-сложного-типа — это избыточная косвенность; не даёт ничего сверх того, что уже даёт паттерн «две фабрики» (вариант E), но добавляет лишний тип; EF-маппинг value object внутри owned entity усложняет конфигурацию.
- **Отклонён**: формальная чистота без практической выгоды.

### Вариант E — Write: фабрики по природе, Read: отдельные DTO + полиморфизм ⭐

**Write-side:** один Domain-класс с приватным конструктором и `internal static` фабриками по каждой природе. Изменения состояния — через wrapper-методы корня агрегата, которые также разделены по природам для Add-операций (`AddCatalogRecipeIngredient`, `AddFreeformRecipeIngredient`). Для Update — один метод (см. ниже §4 Rationale).

**Read-side:** на каждую природу — свой DTO; общий абстрактный record или `interface` как объединяющий тип; дискриминация через атрибуты `[JsonPolymorphic(TypeDiscriminatorPropertyName = "type")]` + `[JsonDerivedType(typeof(...), "...")]` (.NET 8 `System.Text.Json`). Поле `"type"` в JSON-ответе автоматически отображает природу. Та же стратегия применяется к сериализации в jsonb-снепшоты (например, `Dish.PublishedVersionData` — массив `ingredients[]` с дискриминатором у каждого элемента).

**Команды Application-слоя:**

- Для Add — **две команды** на каждую природу (`AddCatalogXxxToYyyCommand`, `AddFreeformXxxToYyyCommand`). Структурная гарантия XOR на уровне типа команды.
- Для Update — **одна команда** (`UpdateXxxCommand`). Метод `Update` в Domain допускает смену природы (catalog ↔ freeform), runtime-проверка XOR в самой сущности через доменную ошибку.

**CHECK-constraint в БД** остаётся как defense-in-depth.

- **Плюсы:** типобезопасность на write (compile-time) и на read (DTO без nullable-капканов); естественная эволюция в специфичные поля у одной природы; совместимость с EF Core (one type — one table, никаких TPH-сюрпризов); полиморфная JSON-сериализация настраивается один раз через атрибуты.
- **Минусы:** объяснимая, но требующая внимания асимметрия «две фабрики и две Add-команды vs один Update»; полиморфная сериализация требует регистрации `[JsonDerivedType]` для каждой природы — при добавлении новой природы можно забыть.
- **Выбран** как комбинация преимуществ остальных вариантов без их характерных минусов.

## 3. Decision (Принятое решение)

В проекте принимается следующий общий принцип работы с дискриминированными объединениями:

> **Доменные DU-сущности (вариативные сущности с дискриминацией по природе) на уровне Domain хранятся в одном классе с фабриками для каждой природы. На уровне Read-DTO — разделяются на отдельные DTO с явным полем-дискриминатором `type`. Это применение CQRS-подхода: write-side оптимизирован под целостность и операции, read-side — под удобство потребления.**

Конкретные правила:

1. **Domain:** один класс на DU-сущность, приватный конструктор, по одной `internal static` фабрике на каждую природу. Имя фабрики содержит имя природы: `CreateFromCatalog`, `CreateFreeform`, `FromTelegram`, `FromInstagram` и т.п.
2. **Update в Domain:** один метод `Update(...)` с runtime-XOR через доменную ошибку. Допускается смена природы.
3. **Команды Application-слоя:** на Add — отдельная команда на каждую природу (`Add<Nature><Entity>Command`). На Update — общая команда.
4. **Read-DTO:** один абстрактный record (или `interface`) и по одному наследнику на природу. Поле-дискриминатор `type` (литерал — имя природы в lowercase, `"catalog"`, `"freeform"`, и т.п.).
5. **JSON-сериализация:** через `[JsonPolymorphic(TypeDiscriminatorPropertyName = "type")]` + `[JsonDerivedType(typeof(...), "...")]`. Регистрация — на базовом типе, один раз для всего проекта.
6. **jsonb-снепшоты** (включая `Dish.PublishedVersionData`): следуют тому же паттерну — массив элементов с дискриминатором у каждого. Сериализатор тот же.
7. **БД:** CHECK-constraint остаётся последней линией защиты XOR — независимо от Domain-фабрик и runtime-проверки в Update.

## 4. Rationale (Обоснование)

1. **Compile-time гарантия инвариантов на стороне записи.** Невозможно создать невалидный объект, не пройдя через одну из фабрик. Невозможно поднять невалидную команду — `Add<Nature>` принимает строго те поля, которые нужны конкретной природе.
2. **Compile-time гарантия на стороне чтения.** Потребитель работает с одним из конкретных DTO — все поля non-nullable (там, где это допустимо доменом). Никаких «пяти nullable-полей-капканов». На клиенте (TypeScript / C# pattern matching) — естественное сужение типа через дискриминатор.
3. **Эволюция: специфичные поля.** Когда у одной природы появятся свои поля (например, `IngredientImageUrl` для catalog) — это естественное расширение конкретного DTO, не раздувание общего.
4. **Асимметрия Add (две команды) и Update (одна) — обоснованна.** Add — это **создание** объекта, момент максимальной строгости (нужно решить, какой природы создаём). Update — это **переход состояния** существующего объекта, где смена природы — допустимая операция, и runtime-проверки в Domain достаточно. Дробление Update на N команд («сменить природу», «обновить amount только catalog», «обновить amount только freeform») усложнит UI без выгоды.
5. **Совместимость с EF Core.** Один тип — одна таблица — никаких TPH-маппингов. Миграции не усложняются, фильтры по природе — это обычные предикаты по дискриминирующей колонке.
6. **Полиморфная сериализация настраивается один раз через атрибуты.** Не требует custom-конвертеров, не зависит от внешних библиотек.
7. **Согласованность с уже существующим разделением DTO.** В проекте `Dish` ≠ `DishCardDto` ≠ `DishDetailDto`. Read-разделение по природе — естественное продолжение того же подхода, а не новый паттерн.

## 5. Consequences (Последствия)

### Positive (Положительные)

- Структурная защита XOR-инвариантов на этапе компиляции для всех write-операций.
- Read-DTO без nullable-полей-капканов — клиентский код становится короче и безопаснее.
- TypeScript-клиенты получают естественный discriminated union с автоматическим сужением типа в `switch`/`match`.
- Эволюция в специфичные для одной природы поля — без рефакторинга общих структур.
- jsonb-снепшоты совместимы с тем же паттерном — сериализатор и десериализатор едины.

### Negative / Trade-offs (Отрицательные / компромиссы)

- Кода чуть больше: две фабрики вместо одной, два Add-DTO/команды вместо одной, два Read-DTO вместо одного. Это разовая надбавка к коду на каждую DU-сущность, не зависит от частоты использования.
- Полиморфная сериализация требует регистрации всех природ через `[JsonDerivedType]`. При добавлении новой природы можно забыть атрибут — сериализация сломается в runtime.
- Документация в Swagger/OpenAPI для полиморфных DTO требует проверки — нужно убедиться, что в схеме отображены все варианты с дискриминатором.

### Areas of Caution (На что обратить внимание)

- При добавлении новой природы в существующую DU-сущность — обязательно обновить **все** места: Domain-фабрика, Domain-метод `Update` (новый ветка XOR), Application-команда `Add<NewNature>`, Read-DTO, атрибут `[JsonDerivedType]` на базовом типе, CHECK-constraint в БД и миграция, схема Swagger-документации. Чек-лист «добавление новой природы DU-сущности» полезно вынести в `08_Разработка-(Development-Guide)` при первом расширении (Этап 5+).
- При reviewing Pull Request с новой DU-сущностью — проверять, что все семь правил §3 соблюдены.
- jsonb-снепшоты — самая хрупкая точка: если сериализатор и десериализатор разойдутся по конфигурации `JsonPolymorphic`, чтение существующих снепшотов сломается. Покрывать unit-тестом round-trip-сериализации каждой природы.

## 6. Future Scope (Будущие направления)

Принцип будет применяться к следующим запланированным DU-сущностям:

- **Этап 5 — `UserSocialLink`** (Users): варианты сети — Telegram / Instagram / VK / website / custom. Каждый вариант имеет свой формат идентификатора и валидации.
- **Этап 6 — `OrderItem`** (Orders): варианты позиции заказа — блюдо / набор блюд / услуга (мастер-класс, доставка). Каждый вариант имеет свою структуру цены и метаданных.
- **Этап 8+ — события и уведомления** (Notifications, Social): естественно вариативный домен — `DishPublishedNotification`, `OrderStatusChangedNotification`, `FollowerAcquiredNotification` и т.п. Здесь сериализация в БД (хранение истории уведомлений) — основной потребитель полиморфного формата.

При появлении каждого нового применения — отдельный узкий ADR (`ADR-NNNN-user-social-link-variants` и т.п.), цитирующий этот ADR как reference и подробно описывающий специфику конкретной сущности.

При появлении в C# нативной поддержки discriminated unions (proposal [Type Unions](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/proposals/unions) в `dotnet/csharplang`) — пересмотр этого ADR: новые применения могут перейти на нативный синтаксис, существующие — либо мигрировать, либо остаться в текущем паттерне с соответствующей пометкой.
