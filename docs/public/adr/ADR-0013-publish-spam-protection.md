# ADR-0013: Защита `Dish.Publish` от спама `DishPublishedEvent` — Domain-инвариант

## 1. Context (Контекст)

`Dish.Publish(...)` — операция перехода блюда в публикуемое состояние с заполнением jsonb-снепшота `PublishedVersionData`, фиксацией `PublishedAt` и поднятием доменного события `DishPublishedEvent`.

На текущем этапе (2) у `DishPublishedEvent` нет ни одного подписчика — событие поднимается «вхолостую» через MediatR `INotificationHandler<DishPublishedEvent>`. На Этапе 5+ событие потенциально вызывает каскад побочных эффектов:

- Рассылка уведомлений подписчикам автора блюда (Notifications).
- Появление блюда в ленте новостей (Social).
- Индексация для каталожного поиска.
- Обновление аналитики «новинки», «недавно опубликовано».

**Проблема.** Без защиты от повторных вызовов `Publish` на уже опубликованном и не изменённом блюде клиент может неконтролируемо инициировать каскад «впустую»:

- Случайный двойной клик «Опубликовать» в UI → две одинаковые цепочки уведомлений.
- Намеренные повторные нажатия как способ «поднять» блюдо в ленте без реальных правок.
- Программные ошибки клиента, отправляющего публикацию в цикле.

**Условия принятия решения.**

- Состояния блюда: `Draft` → `Published` → (`Unpublished` | `Archived`). Из `Archived` обратно нельзя; из `Unpublished` снова можно `Published` (отдельный сценарий — UC-DSH-005 + UC-DSH-004).
- При успешной публикации заполняется `PublishedVersionData (jsonb)`, `PublishedAt`, `PublishedVersionUpdatedAt`. При `Unpublish` / `Archive` всё это обнуляется (`PublishedAt = null`).
- Между двумя `Publish` автор может вносить правки в основные поля карточки/рецепта. Правки обновляют `UpdatedAt` через `MarkAsUpdated`, но **не** обновляют `PublishedVersionData` автоматически — для применения правок в публичной версии необходим явный повторный `Publish`.
- Значит «новая публикация имеет смысл» ровно тогда, когда после последней публикации были правки: `UpdatedAt > PublishedAt`.

Уже существующие в Domain проверки `Publish` (не входят в скоуп этого ADR, описаны для контекста):

- `Status == Archived` → `CannotPublishArchivedDish` (`Error.Conflict`).
- Контентные инварианты: `MainImageRequiredForPublish`, `StepsRequiredForPublish`, `IngredientsRequiredForPublish`, `TimingRequiredForPublish` — все `Error.Conflict`.

## 2. Considered Alternatives (Рассмотренные альтернативы)

### Вариант A — Никакой защиты в Domain

`Publish` всегда выполняется; `DishPublishedEvent` поднимается на каждом вызове. Защита от спама размазана по подписчикам: каждый сервис-consumer внедряет свою дедупликацию, rate-limit, идемпотентный ключ.

- **Плюсы:** Domain максимально простой; никаких временных проверок.
- **Минусы:** Domain теряет инвариант «событие = реальная публикация» — это плохо для будущих подписчиков. Логика дедупликации повторяется в каждом обработчике. Каждый новый подписчик обязан помнить о проблеме и решать её отдельно.
- **Отклонён**: бизнес-инвариант «нет правок → нет повторной публикации» — это бизнес-инвариант, и его место — Domain.

### Вариант B — Проверка `UpdatedAt > PublishedAt` в Domain ⭐

`Publish` отказывает с `DishAlreadyPublished`, если блюдо уже `Status == Published` и `UpdatedAt <= PublishedAt` (несохранённых правок нет).

- **Плюсы:** инвариант на корне агрегата, без дополнительных полей и без отдельной инфраструктуры. Подписчики могут полагаться на то, что событие соответствует реальной публикации.
- **Минусы:** завязка на корректное обновление `UpdatedAt` — любой новый метод-модификатор должен соблюдать соглашение `MarkAsUpdated(utcNow)` / прямого присваивания `UpdatedAt = utcNow`.
- **Выбран ⭐**: текущая реализация в коде Этапа 2.

### Вариант C — Явный флаг `HasUnpublishedChanges`

Boolean-поле на агрегате. Ставится в `true` любым методом-модификатором, в `false` — внутри `Publish`. `Publish` отказывает при `false`.

- **Плюсы:** более явная семантика, прямой ответ на вопрос «есть ли смысл публиковать».
- **Минусы:** избыточная денормализация — `UpdatedAt > PublishedAt` уже даёт ту же информацию. Дополнительное поле в БД и миграция. Те же риски «забыли установить флаг в новом методе», что и в Варианте B, плюс ещё один слой синхронизации.
- **Отклонён**: не даёт ничего сверх Варианта B.

### Вариант D — Защита на уровне Application (rate-limit / idempotency-key)

Application проверяет минимальный интервал между двумя `Publish` для одного блюда, либо требует idempotency-key в запросе.

- **Плюсы:** защищает от двойного клика и ретраев сети; подходит для общего системного механизма идемпотентности (платежи и т.п.).
- **Минусы:** защищает от технических повторов, но не от логически повторяющейся бизнес-операции «нажму ещё раз через 5 минут, может в ленте поднимется». Корневую проблему — «нет смысла публиковать без правок» — не решает. Дублирует Domain-инвариант, не заменяя его.
- **Отклонён как замена Варианту B**, но рассматривается как **дополнительный слой защиты** на будущих этапах (см. §6 Future Scope).

### Вариант E — Подавление события в подписчиках

`Publish` всегда выполняется и поднимает `DishPublishedEvent`. Каждый подписчик внутри своего `INotificationHandler<DishPublishedEvent>` сравнивает `event.PublishedAt` с состоянием своих внутренних данных и подавляет реакцию, если событие повторное.

- **Плюсы:** Domain не несёт инвариант, проще тестировать без проверок времени.
- **Минусы:** логика дедупликации повторяется в каждом обработчике. Domain снова не несёт бизнес-инвариант, что противоречит DDD-принципу. Новые подписчики обязаны помнить о проблеме.
- **Отклонён**: разновидность Варианта A с теми же минусами.

## 3. Decision (Принятое решение)

В методе `Dish.Publish(...)` сохраняется проверка вида:

> Если `Status == DishStatus.Published`, `PublishedAt.HasValue` и `UpdatedAt <= PublishedAt.Value` — операция отказывает с доменной ошибкой `DishesErrors.DishAlreadyPublished` (`Error.Conflict` → HTTP 409).

Конкретно:

1. Защита от спама `DishPublishedEvent` реализована **как Domain-инвариант** в методе `Dish.Publish`.
2. Условие отказа — `Status == Published && PublishedAt.HasValue && UpdatedAt <= PublishedAt.Value`.
3. Доменная ошибка — `DishesErrors.DishAlreadyPublished` (тип `Error.Conflict`). Маппинг в HTTP — `409 Conflict` через `ApiController.MapError`. Расширение `DishesErrors` не требуется — код уже существует.
4. На уровне Application-слоя (UC-DSH-004 PublishDish) **дополнительной защиты не добавляется** на Этапе 2. Application-слой ограничивается передачей результата `Publish` в контроллер.
5. Семантика метода: `Publish` поднимает `DishPublishedEvent` тогда и только тогда, когда фактически произошёл переход в `Published` со сменой `PublishedVersionData` / `PublishedAt`. Подписчики событий вправе полагаться на эту семантику без дополнительной дедупликации (на Этапе 2-4).
6. Будущий механизм принудительной пересборки снепшота админом (Этап 8+) — отдельный метод `RebuildPublishedSnapshot(...)`, **не поднимающий `DishPublishedEvent`** (или поднимающий отдельное событие `DishSnapshotRebuiltEvent`). Это сохранит семантику «`DishPublishedEvent` = реальная публикация».

## 4. Rationale (Обоснование)

1. **Бизнес-инвариант живёт в Domain.** «Нет правок → нет смысла публиковать» — бизнес-условие, не техническая оптимизация. Место такого инварианта — корень агрегата, где он защищён системой типов и не зависит от того, какие слои инфраструктуры стоят сверху.
2. **Не требует дополнительных полей.** `UpdatedAt` и `PublishedAt` уже есть в агрегате — без них агрегат не может корректно работать. Защита от спама становится «бесплатной» сверх существующей инфраструктуры.
3. **Подписчики получают чистый контракт.** В Notifications (Этап 5+) и Social (Этап 5+) обработчики `DishPublishedEvent` могут писать прямую бизнес-логику без своей дедупликации — событие гарантированно соответствует реальной публикации.
4. **Совместимо с будущим админским rebuild.** Сценарий «принудительная пересборка снепшота админом без изменения автором» (например, при массовой миграции структуры snapshot на Этапе 8+) не нарушает инвариант — для него предусматривается отдельный метод `RebuildPublishedSnapshot` без поднятия `DishPublishedEvent`.

## 5. Consequences (Последствия)

### Positive (Положительные)

- Domain-инвариант защищён сейчас, без правок кода на момент принятия ADR.
- Подписчики `DishPublishedEvent` (Этап 5+) могут разрабатываться без своей дедупликации — гарантированная чистота контракта события.
- HTTP-ответ `409 Conflict` с кодом `DISHES.DISH_ALREADY_PUBLISHED` даёт UI понятный сигнал «нет правок, повторная публикация не имеет смысла» — UI может скрыть кнопку «Опубликовать» или показать тёплую подсказку.

### Negative / Trade-offs (Отрицательные / компромиссы)

- Завязка на корректное обновление `UpdatedAt` любым новым методом-модификатором — если кто-то добавит метод, изменяющий поле без `MarkAsUpdated`, защита деградирует тихо (событие будет поднято при пустом изменении).
- На Этапах 5+/6+ может потребоваться дополнительный слой защиты (rate-limit/idempotency-key/дедупликация в consumer'е) — текущая защита покрывает «нет правок», но не «правки слишком близко по времени» и не «двойной запрос из-за сетевой ошибки клиента».

### Areas of Caution (На что обратить внимание)

- При код-ревью любого нового публичного метода на агрегате `Dish` или подчинённых сущностях — убеждаться, что метод корректно обновляет `UpdatedAt` (через `MarkAsUpdated(utcNow)` либо прямое присваивание).
- Желательно покрыть unit-тестом проверкой «после вызова любого публичного метода-модификатора `UpdatedAt` обновляется». Тест становится живой документацией соглашения. Точная форма теста — отдельная задача (см. §6).
- При появлении новых проверок-инвариантов в `Publish` — соблюдать порядок: статусные проверки (`Archived`, `AlreadyPublished`) идут **до** контентных (`MainImageRequiredForPublish`, `StepsRequiredForPublish` и т.п.). Сначала отвечаем на «можно ли в принципе», потом на «достаточно ли данных».

## 6. Future Scope (Будущие направления)

- **Этап 5+ — дедупликация на стороне consumer'ов событий.** При появлении подписчиков `DishPublishedEvent` (рассылка подписчикам автора, индексация и т.п.) — рассмотреть дополнительный слой защиты в самих обработчиках: дедупликация по `(DishId, временное окно T)` и/или минимальный интервал между событиями. Domain-инвариант остаётся **первой линией** защиты (не пропускает событие, если правок нет). Consumer-слой защищает от случаев, когда формально прошедшие Domain-проверку события идут слишком близко по времени друг к другу. Подробности дедупликации (выбор окна, состояние) — отдельный ADR при появлении первого реального подписчика.
- **Этап 6+ — Idempotency-key как общесистемный механизм.** При появлении платежей и других потенциально дублирующихся операций (Publish, оплата, перевод средств) — рассмотреть общесистемный механизм идемпотентности через заголовок `Idempotency-Key` и серверное хранение результата операции по ключу. Это уровень выше Domain — Application/контроллер. Возможен отдельный ADR на Этапе 6+. Защита Publish из этого ADR-0013 остаётся независимой и работает в Domain.
- **Этап 8+ — `RebuildPublishedSnapshot` для админских операций.** Сценарий «массовая пересборка снепшотов админом без поднятия `DishPublishedEvent`» — отдельный метод на агрегате, с собственным сценарием авторизации (`Admin`-only) и отдельным событием (либо без события вообще). Не конфликтует с защитой `Publish` — это другая точка входа.
- **Расширение `DishesErrors` на Этапах 5+/6+.** При появлении нового слоя защиты (например, «минимальный интервал между публикациями») может потребоваться новый код вроде `DishPublishedTooSoon`. На момент принятия ADR расширение не требуется.

## 7. Implementation Reference (Связь с кодовой базой)

  ```csharp
  if (Status == DishStatus.Published
      && PublishedAt.HasValue
      && UpdatedAt <= PublishedAt.Value)
  {
      return Result.Failure(DishesErrors.DishAlreadyPublished);

  ```
