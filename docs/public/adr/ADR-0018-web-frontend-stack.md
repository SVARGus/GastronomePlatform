## 1. Context (Контекст)

Backend завершён в объёме, достаточном для полноценного интерфейса: 78 публичных эндпоинтов в пяти модулях, JWT-аутентификация, документированный контракт. Веб-интерфейс — единственный невыполненный критерий готовности MVP.

Дорожная карта фиксировала выбор технологии как открытый вопрос с двумя кандидатами: ASP.NET Core MVC или Blazor. Оба предполагают, что интерфейс живёт внутри решения и рендерится сервером.

С момента, когда эта формулировка появилась, добавились три обстоятельства, которых она не учитывает. Во-первых, дизайн проработан до уровня брифов с полной дизайн-системой, и его реализация — задача клиентской вёрстки, а не серверных шаблонов. Во-вторых, возникло требование к будущему: интерфейс предполагается со временем вынести в отдельный проект для удобства сопровождения. В-третьих, у разработчика есть практический опыт клиентского стека (React, TypeScript, Tailwind) и параллельно идёт его углублённое изучение, тогда как Razor и Blazor потребовали бы освоения с нуля.

Ограничение по сроку — около двух недель до предзащиты — делает цену неверного выбора высокой: переделка интерфейса в этот срок не помещается.

## 2. Considered Alternatives (Рассмотренные альтернативы)

### Вариант A — ASP.NET Core MVC с Razor-вьюхами

Разметка в `.cshtml`, данные через model binding, интерактивность — точечными вставками JavaScript или TypeScript поверх готового HTML.

**Плюсы:** один язык и одно решение; знакомая экосистема .NET; нет отдельной сборки фронтенда; аутентификация может опираться на серверную сессию.

**Минусы:** интерфейс неотделим от сервера — разметка собирается из C#-модели, и вынос в отдельный проект означает переписывание, а не перенос. Razor — навык, не переиспользуемый за пределами .NET. Реализация проработанной дизайн-системы (компоненты, состояния, анимации) в серверных шаблонах требует больше усилий, чем в компонентной модели.

**Причина отклонения:** прямо противоречит требованию будущей отделимости и не использует имеющийся опыт разработчика.

### Вариант B — Blazor (Server или WebAssembly)

Компонентная модель на C#, разметка в `.razor`.

**Плюсы:** компонентный подход при сохранении единого языка; типизация сквозная — DTO backend используются напрямую; для дипломной работы легко защищается как «целостный .NET-стек».

**Минусы:** Blazor Server держит постоянное соединение и состояние на сервере — отделить интерфейс невозможно по определению. Blazor WebAssembly отделим технически, но остаётся привязанным к .NET-хостингу и экосистеме, то есть требование «вынести для удобства сопровождения» удовлетворяется формально, а не по сути. Обе разновидности — новый для разработчика фреймворк со своей моделью рендеринга и жизненного цикла компонентов.

**Причина отклонения:** цена освоения не окупается в отведённый срок, а выигрыш от единого языка не перевешивает потери в отделимости и переиспользуемости навыка.

### Вариант C — React + TypeScript, собираемый Vite, отдаваемый существующим приложением ⭐

Отдельный каталог с собственной сборкой; результат сборки — статические файлы, которые отдаёт то же ASP.NET-приложение. Взаимодействие с backend — исключительно по HTTP.

**Плюсы:** используется имеющийся опыт; интерфейс с первого дня общается с сервером только через API, поэтому отделение сводится к переносу каталога; один процесс при запуске — сценарий демонстрации не усложняется; общий origin снимает необходимость в CORS на текущем этапе.

**Минусы:** вторая экосистема сборки в проекте; необходимость вручную поддерживать типы, соответствующие DTO backend; аутентификация полностью на клиенте, включая обновление пары токенов.

### Вариант D — Next.js отдельным приложением

Тот же React, но с фреймворком поверх и собственным процессом.

**Плюсы:** отделённость с первого дня; знакомство разработчика с фреймворком; возможности серверного рендеринга.

**Минусы:** серверный рендеринг в предполагаемом режиме использования не нужен — интерфейс задумывается клиентским. Фреймворк добавляет собственный роутинг, соглашения о структуре и второй процесс, который придётся запускать и объяснять при демонстрации. Требует настройки CORS немедленно.

**Причина отклонения:** плата за возможности, которые в этом проекте не используются. Отделимость достигается и в варианте C без дополнительного процесса.

## 3. Decision (Принятое решение)

1. **Интерфейс реализуется как SPA на React с TypeScript.** TypeScript — основной язык клиентской части; JavaScript без типов не используется.
2. **Сборка — Vite.** Клиентская часть живёт отдельным каталогом с собственными `package.json`, `tsconfig.json` и конфигурацией сборщика.
3. **Результат сборки отдаётся существующим ASP.NET-приложением** как статические файлы. Для этого подключаются раздача статики и fallback на оболочку приложения для неизвестных путей — чтобы клиентский роутинг работал при прямом переходе по ссылке.
4. **Взаимодействие с backend — только по HTTP.** Никакой подстановки данных сервером в разметку. Серверные шаблоны не используются; допускается единственная страница-оболочка, содержащая точку монтирования и ничего более.
5. **Обращения к API изолируются в один слой** с типами, повторяющими DTO backend. Базовый адрес API берётся из конфигурации, а не задаётся в коде.
6. **Аутентификация — на клиенте, поверх существующего JWT.** Серверные сессии и cookie-аутентификация не вводятся.
7. **Стилизация — Tailwind**, значения дизайн-системы (палитра, типографика, радиусы, тени) выносятся в конфигурацию и CSS-переменные.

**Правило, обеспечивающее отделимость:** ни одна часть интерфейса не должна зависеть от того, что его отдаёт именно это приложение. Проверка простая — если после переноса каталога на произвольный статический хост и настройки адреса API интерфейс продолжает работать, правило соблюдено.

## 4. Rationale (Обоснование)

1. **Отделимость обеспечивается архитектурой, а не языком.** Разделяет интерфейс и сервер не выбор React против Blazor, а то, получает ли клиент данные по HTTP или их подставляет сервер при рендеринге. Вариант C соблюдает это условие с первого дня; вариант A нарушает по своей природе.

2. **Срок — определяющее ограничение.** Два варианта из четырёх требуют освоения нового фреймворка. При двух неделях до защиты знакомый инструмент снижает риск сильнее, чем единство языка его повышает.

3. **Отложенное решение об отделении не оплачивается заранее.** Вариант D даёт отделённость немедленно, но платой в виде второго процесса, CORS и неиспользуемых возможностей фреймворка. Вариант C оставляет отделение дешёвой операцией, не неся её стоимость сейчас.

4. **Backend не требует изменений.** Проект изначально построен как чистый API: контроллеры возвращают данные, серверного рендеринга нет. Решение опирается на уже существующую границу, а не создаёт новую.

5. **Навык переиспользуется.** Клиентский стек совпадает с рабочим опытом разработчика и с параллельно изучаемым материалом; Razor и Blazor такой отдачи не дают.

## 5. Consequences (Последствия)

### Positive (Положительные)

- Граница между интерфейсом и сервером проходит по HTTP-контракту — единственной точке, которую придётся сохранить при разделении.
- Вынос интерфейса в отдельный репозиторий сводится к переносу каталога, настройке CORS и указанию адреса API; код компонентов не затрагивается.
- Демонстрация остаётся одношаговой: инфраструктура поднимается контейнерами, приложение — одной командой, интерфейс доступен по тому же адресу, что и API.
- CORS не требуется, пока интерфейс отдаётся тем же приложением, — одной задачей и одним источником ошибок меньше.
- Дизайн-система ложится на компонентную модель напрямую: токены в конфигурации, состояния и варианты — в компонентах.

### Negative / Trade-offs (Отрицательные / компромиссы)

- В проекте появляется вторая экосистема сборки со своими зависимостями и конфигурацией.
- Типы ответов API дублируются: DTO описаны на C# и повторяются на TypeScript. Расхождение между ними компилятор не поймает.
- Аутентификация целиком на клиенте: хранение access-токена, отслеживание истечения, обновление пары через refresh, защита маршрутов — всё это предстоит реализовать явно.
- Сознательно теряется преимущество единого языка, которое давали варианты A и B.

### Areas of Caution (На что обратить внимание)

- **Дублирование типов** — держать их в одном месте клиентской части и обновлять при изменении контракта. При росте расхождений рассмотреть генерацию типов из OpenAPI-схемы, которую уже отдаёт Swagger.
- **Соблазн отступить от правила изоляции.** Любая передача данных мимо HTTP — переменная в шаблоне, предварительно отрендеренный фрагмент — незаметно делает отделение невозможным. Это первое, что стоит проверять при просмотре изменений.
- **Хранение токена.** Выбор места хранения влияет на устойчивость к XSS; решение принимается отдельно и осознанно, а не по умолчанию.
- **Fallback-маршрут** должен отдавать оболочку только для неизвестных путей и не перехватывать `/api/*`, иначе запросы к API начнут возвращать HTML.
- **Объём работ.** Полноценный клиент за две недели требует жёсткого ограничения набора страниц; расширение состава — прямая угроза сроку.

## 6. Future Scope (Будущие направления)

- **Вынос интерфейса в отдельный репозиторий** — когда сопровождение единым решением станет неудобным. Потребуется: перенос каталога, политика CORS на backend, конфигурация адреса API, отдельный процесс сборки и развёртывания.
- **Генерация клиентских типов из OpenAPI** — устраняет ручное дублирование DTO. Оправдано при заметном росте числа эндпоинтов, потребляемых интерфейсом.
- **Серверный рендеринг** — при появлении требований к индексации каталога поисковыми системами. На этом этапе имеет смысл вернуться к рассмотрению Next.js, но уже как к обоснованному выбору, а не авансом.
- **Клиентские приложения (Этапы 7+)** — мобильное и десктопное. Слой обращений к API и типы контракта, изолированные согласно решению, могут быть переиспользованы.

## 7. Implementation Reference (Связь с кодовой базой)

На момент принятия решения интерфейс не реализован. Точки касания с существующим кодом:

- `src/GastronomePlatform.WebAPI/Program.cs` — потребуется подключить раздачу статических файлов и fallback-маршрут на оболочку приложения. Раздача статики сейчас не настроена, каталог `wwwroot` отсутствует.
- `src/GastronomePlatform.WebAPI/Program.cs`, конфигурация JWT — параметры токенов, которые клиент обязан учитывать: время жизни access-токена 15 минут, refresh-токена — 30 дней, claim-ы под короткими именами, роль может быть массивом.
- `src/GastronomePlatform.WebAPI/Controllers/**` — источник контракта, по которому описываются типы клиентской части.
- Политика CORS — **не настроена**; понадобится при вынесении интерфейса на отдельный origin, включая запуск dev-сервера сборщика, если он используется на отдельном порту.
