## Context

Этап 3 вводит рекуррентные подписки с автосписанием. В качестве платёжного шлюза выбрана **ЮKassa** — команда знакома с ней по рабочей практике; альтернативные провайдеры требуют времени на изучение.

Существенные особенности ЮKassa, определяющие архитектуру:

- **ЮKassa не держит расписание автосписаний.** После сохранения способа оплаты (`payment_method_id`) каждое последующее списание магазин инициирует **сам**, отправляя запрос на автоплатёж. Значит планировщик списаний должен находиться на нашей стороне.
- **Webhook доставляется асинхронно и негарантированно одноразово** — при недоступности эндпоинта ЮKassa повторяет доставку до 7 раз в течение 24 часов; возможна повторная доставка успешного уведомления.
- **Webhook по умолчанию не подписан HMAC.** ЮKassa рекомендует проверять подлинность по IP-адресу отправителя и перезапросом реального статуса объекта; в кабинете дополнительно доступен signing secret.
- Нужна защита от двойных списаний (ретраи инициации) и от «подвисших» платежей при потерянных уведомлениях.

## Decision

1. **Абстракция шлюза.** ЮKassa скрывается за интерфейсом `IPaymentGateway` (объявлен в Application, реализация `YooKassaPaymentGateway` в Infrastructure). Провайдер остаётся свопаемым, домен от него не зависит.

2. **Две ответственности внутри модуля Subscriptions** (не отдельные микросервисы):
   - **Сборщик рекуррентов** — фоновая задача, запускается раз в сутки. Выбирает подписки к списанию по `NextBillingAt`, инициирует автоплатежи через `IPaymentGateway`, пишет `SubscriptionPayment(Pending)`. Статус подписки **не двигает**.
   - **PaymentService (webhook-обработчик)** — принимает уведомления ЮKassa, верифицирует, идемпотентно применяет переход машины состояний, публикует интеграционные события.

3. **Хостинг сборщика.** Внешне-триггеримый console-host (Windows Task Scheduler / cron / GitHub Actions schedule), по аналогии с рабочей практикой. Если потребуется встроенная надёжность (персистентное расписание, ретраи, дашборд) — Hangfire / Quartz.NET.

4. **Привязка способа оплаты.** Для платных офферов `payment_method_id` сохраняется из первого реального списания (`Purpose = Initial`). Для `Trial`/`Scheduled` — проверочное списание 1 ₽ (`Purpose = Verification`) с возвратом; рассматривается привязка на нулевую сумму как альтернатива без денежного движения.

5. **Одностадийность.** Автоплатежи создаются с `capture: true` — платёж идёт сразу в `succeeded`, без промежуточного `waiting_for_capture`.

6. **Идемпотентность на двух концах:**
   - исходящая — заголовок `Idempotence-Key`, детерминированный от `(SubscriptionId + период)`, защищает от двойного списания при ретраях инициации;
   - входящая — UNIQUE на `SubscriptionPayment.GatewayTransactionId`, повтор webhook = no-op.

7. **Верификация webhook.** Проверка IP-диапазонов ЮKassa + перезапрос реального статуса (`GET /payments/{id}`); опционально signing secret. Тело уведомления доверенным не считается. Эндпоинт быстро возвращает HTTP 200, тяжёлая обработка — в фон.

8. **Reconciliation.** Сборщик при прогоне дочитывает статус подвисших `Pending` через `GET /payments/{id}` и синхронизирует БД — страховка на случай потерянного webhook.

9. **Антифрод — по правилам шлюза, своё не строим.** Обработка `Failed` ветвится по `cancellation_details.reason` ЮKassa: отказ по бизнес-причине (недостаточно средств, лимит) → обычная неудача → dunning-ретраи; отказ с фрод-причиной → немедленно `AutoRenew = false`, `RecurringDisabledReason = Antifraud`, без ретраев. Текущий оплаченный период сохраняется.

10. **Возврат проверочного списания — синхронно.** Возврат 1 ₽ инициируется в обработчике успешного `Verification`-платежа (`POST /refunds`); завершение приходит своим webhook `refund.succeeded` и обрабатывается идемпотентно. Касается только пути с 1 ₽; у привязки на нулевую сумму возвращать нечего.

11. **Юридическое согласие на рекуррент ≠ техническая привязка.** Согласие образуют: оферта/пользовательское соглашение с раскрытыми условиями автосписаний (сумма, период, автопродление, способ отписки), явный акт согласия на форме (чек-бокс), 3-D Secure на первом платеже, обязательный механизм отписки. Хранение: **термы** — в `SubscriptionAgreement` (версионируемый снепшот) с якорем акта (`AcceptedAt` + версия + хеш); **полный контекст акта** (IP, User-Agent, метод) — событием `ConsentGiven`/`SubscriptionCreated` в бизнес-лог. Отдельный `ConsentLog` не заводим (сценарий B): акт согласия — частный случай юридически значимого действия, которое и так пишет бизнес-лог; версионирование оферты служит журналом согласий.

## Consequences

**Положительные:**

- Провайдер свопаем за `IPaymentGateway`; переход на другой шлюз или вынос платежей в микросервис не трогает домен (паттерн `IAuthUserService` / `IFileStorage`).
- Чёткое разделение «инициатор списания» и «обработчик ответа» — разные жизненные циклы, по отдельности тестируются.
- Двойная идемпотентность защищает от дублей списаний и повторных уведомлений.
- Reconciliation страхует от ненадёжной доставки webhook.

**Отрицательные / компромиссы:**

- Нужен внешний планировщик (или Hangfire) — дополнительная точка входа/инфраструктуры.
- Каденс сборщика (раз в сутки) ограничивает гранулярность ретраев сутками; интервалы мельче потребуют более частого прогона.
- Webhook ненадёжен — обязателен reconciliation-путь, иначе подвисшие `Pending`.
- Операционно: автоплатежи в боевом магазине ЮKassa включаются по запросу менеджеру (в тестовом — доступны сразу).

**Зависимости (сквозные):**

- **Бизнес-лог** (TODO) — append-only аудит юридически значимых действий; принимает события согласия и операций над подпиской. Subscriptions только эмитит события.
- **Лог активностей** (Этап 4+) — фиксация использования услуг; доказательная база для возврата «не пользовался».

**Открытые вопросы:**

- Способ привязки: проверочное списание 1 ₽ против привязки на нулевую сумму — решается на этапе реализации за `IPaymentGateway`, по поддержке нужных методов. По умолчанию 1 ₽.
- Финальные формулировки оферты и пользовательского соглашения — выверить с юристом.
