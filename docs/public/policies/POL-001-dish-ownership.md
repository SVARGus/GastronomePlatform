## 1. Назначение политики

Политика определяет, какие пользователи имеют право модифицировать блюдо (`Dish`)
и его составные части (`Recipe`, `RecipeStep`, `RecipeIngredient`, `Timing`, `Yield`, `Nutrition`, `DishCategory`, `DishTag`).

Эта политика — **единый источник истины** для авторизации операций модификации в модуле Dishes.
Use Cases ссылаются на политику, а не дублируют правила в каждом отдельном UC.

---

## 2. Применимость

### 2.1. UC, использующие политику

Политика применяется ко всем Command-сценариям модуля Dishes, которые изменяют состояние существующего блюда:

- `UC-DSH-002` — Обновить публичную карточку блюда
- `UC-DSH-003` — Обновить рецепт
- `UC-DSH-004` — Опубликовать блюдо
- `UC-DSH-005` — Снять блюдо с публикации
- `UC-DSH-006` — Архивировать блюдо
- `UC-DSH-007` — Изменить категории блюда
- `UC-DSH-008` — Изменить теги блюда
- `UC-DSH-009` — Установить диетические метки блюда
- `UC-DSH-010` — Установить историческое описание блюда
- `UC-DSH-020..023` — Управление шагами рецепта
- `UC-DSH-030..033` — Управление ингредиентами рецепта
- `UC-DSH-040..042` — Установка тайминга, выхода, КБЖУ

### 2.2. UC, НЕ использующие политику

- `UC-DSH-001 CreateDish` — создание нового блюда. Политика не применяется к созданию;
  ограничение на создание задаётся отдельно (любой авторизованный пользователь, кроме Guest).
- Чтения (`UC-DSH-050..064`) — у них собственные правила доступа.
- Админские UC (`UC-DSH-101+`) — у них прямые ограничения на роль `Admin`.

## 3. Правило (Этап 2)

Действие разрешено, если выполняется хотя бы одно из следующих условий:

### 3.1. Условие «Author»

`Dish.AuthorUserId == ActorUserId`

И блюдо не принадлежит платформе (см. таблицу `DishOwnership`, появляется на Этапе 4+).
На Этапе 2 это условие сводится к простому сравнению `AuthorUserId == ActorUserId`,
поскольку таблицы `DishOwnership` ещё не существует.

### 3.2. Условие «Admin»

Пользователь имеет роль `Admin` (из `PlatformRoles`).

---

## 4. Дополнительные ограничения по статусам

Помимо проверки прав, выполнение действия зависит от текущего состояния блюда.

### 4.1. По полю `Dish.Status`

| Status | Author | Admin |
|--------|--------|-------|
| `Draft` | Может модифицировать всё | Может модифицировать всё |
| `Published` | Может модифицировать «рабочую копию» (основные таблицы); публичная версия (`PublishedVersionData`) обновляется только через явный `UC-DSH-004 Publish` | Может модифицировать всё, включая прямую перезапись `PublishedVersionData` |
| `Unpublished` | Может модифицировать всё (как Draft); может вернуть в Published через `UC-DSH-004` | Может модифицировать всё |
| `Archived` | **Запрещено всё** | Может модифицировать (для восстановления) |

### 4.2. По полю `Dish.ModerationStatus` (Этап 8+)

| ModerationStatus | Author | Moderator/Admin |
|------------------|--------|-----------------|
| `Approved` | Может модифицировать (см. правила выше по `Status`) | Может всё |
| `Pending` | Может модифицировать «рабочую копию», но не может публиковать | Может одобрить/отклонить |
| `Rejected` | Может модифицировать «рабочую копию»; повторная публикация требует прохождения модерации | Может всё |
| `Flagged` | Может модифицировать «рабочую копию» | Может всё |

На Этапе 2 `ModerationStatus` всегда `Approved` по умолчанию, эта секция фактически не активна
до Этапа 8+.

## 5. Расширения политики (будущие этапы)

### 5.1. Этап 4+ — Platform Ownership

Появляется таблица `DishOwnership` с двумя типами владения:

- `AuthorOwned` (по умолчанию) — действует базовое правило (см. п. 3).
- `PlatformOwned` — блюдо передано платформе (например, по результатам конкурса).

Для `PlatformOwned` блюд:
- **Условие «Author» не применяется** — автор больше не может модифицировать.
- Модифицировать может только `Admin`.
- Атрибуция «Автор: ...» сохраняется в карточке как декоративная подпись.

Реализация — отдельный шаг проверки в начале политики:

```
if (await IsPlatformOwned(dishId)) → допускать только роль Admin/Moderator
```

### 5.2. Этап 8+ — Роль Moderator

В `PlatformRoles` добавляется роль `Moderator`. Появляется в условии «Admin» наравне с админом.

Использование:
- Модерация контента (UC-DSH-150..153).
- Помощь в массовых действиях с тегами и категориями.
- Управление спорным контентом без полных админских привилегий.

### 5.3. Этап 8+ — Условие «Support с открытым тикетом»

Появляется роль `Support` и модуль обращений. Условие:

```
Пользователь имеет роль Support
И существует открытый тикет, связанный с этим dishId
```

Используется для технической поддержки авторов («не могу опубликовать», «не сохраняется фото»).
Поддержка получает временный доступ только к блюдам, по которым есть активные обращения.

## 6. Реализация в коде

### 6.1. Подход

На Этапе 2 политика реализована **прямой проверкой в Command Handler-ах модуля Dishes**, без выделения в отдельный сервис или ASP.NET `AuthorizationHandler<>`. Обоснование выбора:

- **Изоляция Application от ASP.NET.** Стандартный `IAuthorizationService.AuthorizeAsync` требует `ClaimsPrincipal` как параметр — это бы протекало в Application слой типом из `System.Security.Claims`. Прямая проверка через `ICurrentUserService.UserId` сохраняет Application абстрагированным от транспорта.
- **YAGNI.** Отдельный сервис вида `IDishOwnershipPolicy.AuthorizeModificationAsync(...)` пока не оправдывается объёмом логики — Этап 2 проверяет одно условие (`AuthorUserId == ActorUserId`). Введение сервиса целесообразно, когда добавятся PlatformOwnership (Этап 4+), статусные ограничения с разной логикой по статусу, проверка ролей `Admin`/`Moderator`.
- **ASP.NET-механизм `AuthorizationHandler<>`** появится при первом read-only UC, где fetch ресурса выполняется в контроллере и `ClaimsPrincipal` естественно доступен через `User` (например, UC-DSH-053 GetMyDrafts с фильтрацией по владельцу).

### 6.2. Структура реализации (Этап 2)

В каждом Command Handler-е, который модифицирует существующее блюдо:

```csharp
public async Task<Result> Handle(UpdateDishCardCommand request, CancellationToken ct)
{
    Dish? dish = await _dishRepository.GetByIdAsync(request.DishId, ct);
    if (dish is null)
    {
        return DishesErrors.DishNotFound;
    }

    var actorUserId = _currentUser.UserId!.Value;
    if (dish.AuthorUserId != actorUserId)
    {
        return DishesErrors.NotDishOwner;
    }

    // ... применение изменений ...
}
```

`_currentUser.UserId!.Value` корректно: на эндпоинтах с модификацией применяется политика `AuthorizationPolicies.VALID_ACTOR` (`[Authorize(Policy = VALID_ACTOR)]`), которая гарантирует наличие claim `sub`, парсящегося в `Guid`.

### 6.3. Эволюция в отдельный сервис

При появлении любого из следующих условий — выделяем `IDishOwnershipService` (или `IDishAuthorizationService`) в `Dishes.Application/Authorization/`:

- Появляется PlatformOwnership (Этап 4+) — таблица `DishOwnership` с типом владения, проверка через БД-запрос.
- Появляется `Admin` / `Moderator` / `Support` (Этап 8+) — несколько источников разрешения с приоритетом.
- Появляются статусные ограничения, которые нужно проверять отдельно от ownership (например, `Archived` нельзя редактировать никому, кроме `Admin`).

Сервис будет инжектироваться в Handler-ы. Текущая прямая проверка заменяется на вызов `_ownership.AuthorizeModificationAsync(dish, _currentUser, ct)`.

### 6.4. Ошибки

| Код | HTTP | Условие |
|-----|------|---------|
| `DISHES.DISH_NOT_FOUND` | 404 | Блюдо не существует |
| `DISHES.NOT_DISH_OWNER` | 403 | `ActorUserId ≠ AuthorUserId` (на Этапе 8+ — и не `Admin`, не `Moderator`) |
| `DISHES.DISH_ALREADY_ARCHIVED` | 403/409 (TBD) | Этап 8+ — блюдо архивировано, изменять может только `Admin` |
| `DISHES.PLATFORM_OWNED` | 403 | Этап 4+ — блюдо принадлежит платформе |

Коды ошибок — в `Dishes.Domain/Errors/DishesErrors.cs`.

## 7. Тесты политики

Минимальный набор unit-тестов для `DishOwnershipPolicy`:

- Author может модифицировать своё блюдо в Draft → разрешено
- Author может модифицировать своё блюдо в Published → разрешено
- Author НЕ может модифицировать чужое блюдо → 403 NOT_OWNER
- Author НЕ может модифицировать своё блюдо в Archived → 403 ARCHIVED
- Admin может модифицировать чужое блюдо в любом статусе → разрешено
- Admin может модифицировать своё блюдо в Archived → разрешено
- Несуществующее блюдо → 404 NOT_FOUND
- (Этап 4+) PlatformOwned + Author → 403 PLATFORM_OWNED
- (Этап 4+) PlatformOwned + Admin → разрешено

Тесты пишутся в `Dishes.UnitTests/Authorization/DishOwnershipPolicyTests.cs`.
