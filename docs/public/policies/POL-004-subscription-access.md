## 1. Назначение политики

Политика определяет два связанных вопроса:

1. **Кто может выполнять операции над подпиской** (`UserSubscription`) — оформление, отмена, реактивация, смена способа оплаты, просмотр.
2. **Кто получает доступ к услуге** (`FeatureGrant`), которую предоставляет подписка, — для проверок из других модулей (Dishes, Orders и т.д.).

Эта политика — **единый источник истины** для авторизации в монетизации. Use Cases ссылаются на политику через `Authorization: POL-004`, а не дублируют правила.

## 2. Применимость

### 2.1. UC, использующие политику

Операции над подпиской (проверка владения / роли):

- `UC-SUB-020` — Оформить подписку (eligibility роль-гейта)
- `UC-SUB-021` — Просмотреть свою подписку (владение)
- `UC-SUB-022` — Отменить автопродление
- `UC-SUB-023` — Реактивировать подписку
- `UC-SUB-024` — Сменить способ оплаты
- `UC-SUB-025` — Сменить план (Этап 4+)
- `UC-SUB-070..071` — Возвраты (Этап 4+)

Проверка доступа к услуге (резолвер грантов) — вызывается из любого модуля:

- `UC-SUB-050` — Проверить доступ к фиче
- `UC-SUB-051` — Получить эффективные гранты пользователя

Действующие потребители резолвера в других модулях:

| UC-потребитель | Проверяемый грант | Поведение при отказе |
|----------------|-------------------|----------------------|
| `UC-DSH-052` — Получить рецепт блюда | `FullRecipes` | `403 DISHES.PREMIUM_REQUIRED` |
| `UC-DSH-056` — Пересчитать ингредиенты на порции | `PortionCalculator` | `403 DISHES.PREMIUM_REQUIRED` |

Оба проверяют грант только для стороннего читателя: автор блюда и `Admin` проходят без обращения к резолверу (POL-001). Собственный код ошибки модуля-потребителя вместо `SUBS.FEATURE_NOT_GRANTED` — осознанно: отказ формулируется в терминах запрошенного ресурса, а не подсистемы монетизации.

### 2.2. UC, НЕ использующие политику

- Каталожные admin-UC (`UC-SUB-001..010`) — прямое ограничение на роль `Admin`.
- Витринные чтения (`UC-SUB-040..042`) — публичны (каталог планов виден всем).
- Системные сценарии (`UC-SUB-200..208`) — выполняются в System-контексте, пользовательская авторизация неприменима.

## 4. Правила (Этап 3)

### 4.1. Просмотр подписки

Разрешено Owner своей подписки **или** Admin. Чужую подписку обычный пользователь не видит → `FORBIDDEN_NOT_OWNER`.

### 4.2. Оформление (Subscribe)

Разрешено любому аутентифицированному пользователю при условиях:

- **Покупочный роль-гейт.** Если у Base-плана задан `RequiredRole` (порог «не ниже роли») — проверка eligibility через `IRoleEligibilityService` (KYC делегируется Users; на Этапе 3 — заглушка). Не подтверждено → `FORBIDDEN_ROLE_REQUIRED`. Проверяется только при **покупке/апгрейде**, не на продлении. У `AddOn`-планов покупочного роль-гейта нет.
- **Мульти-слот.** У пользователя допускается ≤1 активной **Base**-подписки. Повторное оформление Base при наличии активной → `ALREADY_HAS_BASE` (на Этапе 3 смены плана нет; апгрейд = отмена + новое оформление). `AddOn`-подписок несколько, но не дубль одного и того же AddOn.
- **Покупаемость оффера.** Оффер должен быть `IsPurchasable && IsActive` и в окне доступности (`AvailableFrom`/`AvailableUntil`). Иначе → `OFFER_NOT_PURCHASABLE`.

### 4.3. Отмена / реактивация / смена способа оплаты / смена плана

Разрешено только Owner **или** Admin → иначе `FORBIDDEN_NOT_OWNER`.

### 4.4. Доступ к услуге (резолвер грантов)

Грант действует, если он входит в **эффективный набор**, собираемый как **объединение по всем активным подпискам** пользователя (Base + AddOn):

```
эффективные гранты =
    ∪ грантов всех активных подписок (Base + AddOn) через PlanGrant
  ∪ гранты добавляющих промо (PromotionGrant, IsGrant = true), таргет которых матчит пользователя
  − гранты убирающих промо (PromotionGrant, IsGrant = false)
  затем фильтр гейта усвоения
```

- Промо фильтруются по сроку (`Promotion.From..To`) и активности.
- **Гейт усвоения** (отдельно от покупочного): грант остаётся, если у него нет `RequiredRole` в реестре усвоения ИЛИ роль у пользователя есть. Роль-привязанные гранты (продвижение/реклама → `Chef`) без роли инертны; потребительские — агностичны.
- **Квота:** для квотовых грантов (`PromotionAdvanced`) возвращается **сумма** `PlanGrant.Quantity` по активным подпискам (`null` = безлимит). Учёт расхода — в Dishes.
- **Грантовый floor:** при отсутствии активной Base резолвер подмешивает базовый набор (период привлечения, флаг конфигурации).

Нет гранта → `FEATURE_NOT_GRANTED`.

### 4.4a. Карательное понижение роли

Понижение роли (за нарушения) подписку **не отменяет**, возврат **не делает**, продление **не блокирует**. Роль-привязанные гранты гаснут автоматически через гейт усвоения (4.4); потребительские гранты и подписка работают. Отдельной обработки не требует — решается в резолвере. UC-SUB-208 при понижении лишь логирует/уведомляет.

### 4.4b. Роль — предусловие подписки, а не её следствие

Зависимость между ролью и подпиской **односторонняя**: роль открывает доступ к покупке, но подписка ролью не управляет.

- **Роль → подписка (есть).** Покупочный роль-гейт §4.2: чтобы оформить план с заданным `RequiredRole`, пользователь должен заранее удовлетворять критериям подъёма по роли — подтверждённые данные в личном кабинете (например, ИНН самозанятого), подписанные соглашения, документы. Проверка делегируется `IRoleEligibilityService` (KYC на стороне Users, §5.1).
- **Подписка → роль (нет).** Оформление, отмена и истечение подписки **не отзывают роль** пользователя. Ни один статусный переход `UserSubscription` не порождает понижения роли, и такого сценария в модели не предусмотрено.

**Почему это безопасно.** Доступ к платным возможностям определяется не ролью, а эффективным набором грантов (§4.4). Гранты резолвятся только по активным подпискам с непросроченным `CurrentPeriodEnd`, поэтому по окончании оплаченного периода доступ прекращается автоматически — независимо от того, какие роли остались у пользователя. Роль, полученная при активации подписки, остаётся маркером и ключом доступа не является.

**Практическое следствие.** Фоновое истечение подписки (§4.5) ограничивается приведением статуса в актуальное состояние и не обращается к управлению ролями. Обработчики доменных событий подписки роль не понижают.

### 4.5. Системные операции

Инициация списаний, переходы по webhook, понижение по Fallback, истечение — только System-контекст. Webhook верифицируется по IP-диапазонам ЮKassa + перезапросу статуса (см. ADR-0017 `recurring-payments-yookassa`).

### 4.6. Возвраты (Этап 4+)

Запрос инициирует Owner, решение принимает Admin. На Этапе 3 — ручная обработка админом; на Этапе 4+ — с опорой на лог активностей (доказательство использования).

### 4.7. Аудит

Все юридически значимые операции (оформление, продление, отмена, смена плана) эмитят события для **бизнес-лога** (сквозной модуль). Контекст акта согласия пишется туда же (сценарий B, см. доменную модель).

## 5. Расширения политики (будущие этапы)

### 5.1. Этап 6 — KYC-eligibility

`IRoleEligibilityService` наполняется реальной логикой: бизнес-план доступен к покупке только при подтверждённой роли Chef/Restaurant (ИНН, реквизиты, договор, документы об образовании — на стороне Users/verification).

### 5.2. Этап 8+ — Роль Support

Появляется роль `Support` и модуль обращений. Условие: пользователь имеет роль `Support` И существует открытый тикет, связанный с этой подпиской. Даёт временный доступ к управлению подпиской для техподдержки (например, ручной возврат, перепривязка карты).

## 6. Реализация в коде

### 6.1. Структура

Две точки, намеренно разнесённые по слоям.

Авторизация операций над подпиской — внутреннее дело модуля, поэтому контракт живёт в Application:

```csharp
namespace GastronomePlatform.Modules.Subscriptions.Application.Authorization;

public interface ISubscriptionAccessPolicy
{
    Task<Result> AuthorizeOperationAsync(
        Guid subscriptionId,
        Guid actorUserId,
        IReadOnlyCollection<string> actorRoles,
        CancellationToken ct = default);
}
```

Резолвер грантов вызывается **из других модулей**, поэтому его контракт вынесен в Domain — модуль-потребитель ссылается на Domain провайдера, а не на его Application. Тот же приём, что у `IAuthUserService` модуля Auth:

```csharp
namespace GastronomePlatform.Modules.Subscriptions.Domain.Contracts;

public interface ISubscriptionAccessService
{
    Task<bool> HasFeatureAsync(
        Guid userId,
        FeatureGrant grant,
        CancellationToken ct = default);

    Task<IReadOnlyCollection<FeatureGrant>> GetEffectiveGrantsAsync(
        Guid userId,
        CancellationToken ct = default);
}
```

Реализации обеих точек — в `Subscriptions.Application.Authorization`.

### 6.2. Использование в Handler'е

```csharp
public async Task<Result> Handle(CancelSubscriptionCommand cmd, CancellationToken ct)
{
    var authResult = await _accessPolicy.AuthorizeOperationAsync(
        cmd.SubscriptionId,
        _currentUser.UserId,
        _currentUser.Roles,
        ct);

    if (authResult.IsFailure)
        return authResult;

    // загрузка агрегата и применение изменений
}
```

### 6.3. Ошибки

| Код | HTTP | Условие |
|-----|------|---------|
| `SUBS.NOT_FOUND` | 404 | Подписка не существует |
| `SUBS.FORBIDDEN_NOT_OWNER` | 403 | ActorUserId ≠ UserId и нет роли Admin |
| `SUBS.FORBIDDEN_ROLE_REQUIRED` | 403 | План требует роль, не подтверждённую у пользователя |
| `SUBS.FEATURE_NOT_GRANTED` | 403 | Эффективный набор грантов не содержит требуемую услугу |
| `SUBS.ALREADY_HAS_BASE` | 409 | У пользователя уже есть активная Base-подписка (мульти-слот: ≤1 Base) |
| `SUBS.OFFER_NOT_PURCHASABLE` | 400 | Оффер недоступен к покупке (флаги/окно доступности) |
| `SUBS.CANNOT_CANCEL_IN_STATUS` | 409 | Отмена запрошена для подписки не в статусе `Trialing`/`Active` |
| `SUBS.CANNOT_EXPIRE_IN_STATUS` | 409 | Истечение запрошено для подписки не в статусе `Trialing`/`Active`/`Canceled` |

Таблица ограничена ошибками, относящимися к области этой политики: авторизация операций над подпиской, покупочные инварианты и статусные переходы её жизненного цикла. Инварианты каталога (`SubscriptionPlan`, `PlanPrice`) и журнала платежей сюда не входят — их полный перечень ведётся в коде.

Коды ошибок — в `SubscriptionsErrors.cs` модуля Subscriptions. Префикс `SUBS.` зафиксирован в `08_Разработка-(Development-Guide).md` §4 (таблица префиксов модулей).

HTTP-статусы соответствуют централизованному маппингу `ErrorType → HTTP` в `ApiController.MapError`: `Validation → 400`, `NotFound → 404`, `Forbidden → 403`, `Conflict → 409`. Отдельного типа для `422 Unprocessable Entity` в проекте нет, поэтому отказ в покупаемости оффера отдаётся как `400`.

## 7. Тесты политики

Минимальный набор unit-тестов:

- Owner видит свою подписку → разрешено
- Owner НЕ видит чужую подписку → 403 NOT_OWNER
- Admin видит/изменяет любую подписку → разрешено
- Оформление Base-плана без RequiredRole любым пользователем → разрешено
- Повар покупает потребительский Base (порог не мешает высокой роли) → разрешено
- Оформление роль-гейтованного Base без подтверждённой роли → 403 ROLE_REQUIRED
- Второй активный Base при уже активном → 409 ALREADY_HAS_BASE
- Покупка непокупаемого оффера (`IsPurchasable = false`) → 422 NOT_PURCHASABLE
- `HasFeatureAsync` для активной подписки с агностик-грантом → true
- Гранты двух активных подписок (Base + AddOn) объединяются (union) → true для обоих
- Роль-привязанный грант без роли (карательное понижение) → false (инертен)
- Роль-привязанный грант с ролью → true
- Квотовый грант: сумма `Quantity` по двум подпискам → корректная сумма

Тесты — в `tests/Unit/GastronomePlatform.Subscriptions.UnitTests/Application/Authorization/`.

Отдельно проверяется правило §4.4b: истечение подписки (`UserSubscription.Expire`) не обращается к управлению ролями, а после окончания периода резолвер перестаёт отдавать гранты независимо от статуса записи и от набора ролей пользователя.
