## 1. Где какая документация живёт

Проект разделяет документацию на два типа источников:

| Источник | Назначение | Аудитория | Кириллица в именах |
|----------|-----------|-----------|---------------------|
| **GitHub Wiki** | Высокоуровневые, относительно стабильные обзорные документы: описание проекта, архитектура, дорожная карта, инструкция по запуску, общие гайды | Внешние читатели (комиссия по диплому, потенциальные контрибьюторы) | Да, разрешена |
| **`docs/public/`** в репозитории | Рабочие технические артефакты, эволюционирующие вместе с кодом: спецификации доменных моделей, use cases, ADR, политики, шаблоны | Разработчик(и) проекта | Нет, только латиница |

### Правило разграничения

- В **Wiki** — что и зачем (концепции, обзоры, гайды).
- В **`docs/public/`** — как именно (детали моделей, контракты, конкретные сценарии).

### Аргументы в пользу `docs/public/` для технической документации

1. **Версионирование вместе с кодом.** PR, меняющий `Dish.cs`, в том же коммите обновляет `docs/public/modules/dishes/domain-model.md`. Документация не разъезжается с реальностью.
2. **Code review.** Изменения в спецификациях проходят те же ревью, что и код.
3. **Поиск через IDE.** Visual Studio находит нужный документ рядом с классами.
4. **Латиница в путях.** Совместимость git между Windows и Linux, ссылки в Markdown без URL-encoding, готовность к будущей публикации статичного сайта (MkDocs / Docusaurus).

## 2. Структура папки `docs/public/`

```
docs/
├── public/
│   ├── README.md                      ← оглавление docs/public
│   ├── documentation-conventions.md   ← этот документ
│   │
│   ├── DocTemplates/                  ← шаблоны для новых документов
│   │   ├── UseCase-template.md
│   │   └── (в будущем: ADR-template.md, DomainModel-template.md, ...)
│   │
│   ├── adr/                           ← Architectural Decision Records (общие, на уровне проекта)
│   │   ├── README.md                  ← оглавление ADR
│   │   └── ADR-0001-modular-monolith.md
│   │
│   ├── policies/                      ← политики авторизации
│   │   ├── README.md
│   │   ├── POL-001-dish-ownership.md
│   │   └── ...
│   │
│   └── modules/                       ← документация по модулям
│       ├── README.md                  ← оглавление модулей
│       │
│       ├── auth/
│       │   ├── domain-model.md
│       │   └── use-cases/
│       │       ├── README.md          ← таблица UC модуля
│       │       ├── UC-AUT-001-Register.md
│       │       └── ...
│       │
│       ├── users/
│       │   ├── domain-model.md
│       │   └── use-cases/
│       │
│       ├── dishes/
│       │   ├── domain-model.md
│       │   └── use-cases/
│       │
│       └── media/
│           ├── domain-model.md
│           └── use-cases/
```

`README.md` в каждой папке создаётся **по мере необходимости** — когда документов в папке становится больше двух и нужна навигация.

## 3. Соглашения по именованию файлов

### 3.1. Общие правила для `docs/public/`

- **Латиница**, lowercase, **kebab-case** для слов: `domain-model.md`, `event-bus-setup.md`.
- Без цифровых префиксов вида `01_`, `02_` — используем `README.md` в каждой папке как оглавление с нужным порядком.
- Без указания этапа в имени файла. Документ `domain-model.md` описывает **текущее состояние модели**, а не «версию на Этапе 2». При расширении модели на Этапе 5 — обновляется тот же файл, не создаётся новый. История изменений живёт в `git log`.

### 3.2. Шапка документа

В каждом документе вверху — стандартная шапка с метаданными:

```markdown
# Название документа

> **Версия:** 1.0 (Этап 2)
> **Последнее изменение:** YYYY-MM-DD
> **Статус:** Проектирование / В разработке / Реализовано
> **Связанные документы:** ссылки на Wiki и другие docs
```

### 3.3. Wiki — соглашения остаются как есть

В Wiki сохраняется текущая схема: цифровые префиксы и кириллица (`01_Описание-проекта.md`, `02_Архитектура.md`). Не меняем.

## 4. Соглашения по ID артефактов

### 4.1. Use Cases — `UC-<MODULE>-<NNN>`

Формат идентификатора: `UC-<MODULE_PREFIX>-<NNN>`.

**Префиксы модулей** (3 буквы, строго фиксированный список — добавление новых только через PR):

| Модуль | Префикс |
|--------|---------|
| Auth | `AUT` |
| Users | `USR` |
| Dishes | `DSH` |
| Media | `MED` |
| Subscriptions | `SUB` |
| Notifications | `NTF` |
| Social | `SOC` |
| Orders | `ORD` |

**Диапазоны номеров:**

| Диапазон | Назначение |
|----------|-----------|
| 001–099 | Основные пользовательские сценарии |
| 100–199 | Админские сценарии (модерация, управление справочниками) |
| 200–299 | Системные сценарии (фоновые задачи, обработчики событий) |
| 300–399 | Зарезервировано |

**Примеры:**

```
UC-AUT-001-Register
UC-DSH-001-CreateDishDraft
UC-DSH-002-PublishDish
UC-DSH-101-AdminApproveDish      (админский)
UC-DSH-201-RecalculateRating     (системный, обработчик события)
UC-MED-001-UploadFile
UC-MED-201-CleanupOrphanedFiles  (фоновая задача)
```

**Имя файла:** `UC-<MODULE>-<NNN>-<PascalCaseName>.md`. Это позволяет находить файл по ID и одновременно понимать его содержание.

**ID не переиспользуется.** Если UC отменён — ID помечается как `Deprecated` в оглавлении и в файле, но остаётся занятым. Новые UC получают следующие свободные номера.

### 4.2. ADR (Architecture Decision Records) — `ADR-<NNNN>`

Архитектурные решения, принятые на уровне проекта. Описание: контекст → варианты → выбранное решение → последствия.

**Формат ID:** `ADR-<NNNN>` (4 цифры, чтобы хватило надолго).

**Имя файла:** `ADR-<NNNN>-<kebab-case-title>.md`.

**Примеры:**

```
ADR-0001-modular-monolith.md
ADR-0002-clean-architecture.md
ADR-0003-cqrs-with-mediatr.md
ADR-0004-postgresql-as-primary-storage.md
ADR-0005-rabbitmq-for-event-driven.md
ADR-0006-dish-recipe-single-aggregate.md
ADR-0007-media-ownership-and-data-residency.md
```

**ADR — иммутабельны.** Если решение пересмотрено — создаётся новый ADR со статусом `Accepted`, старый — со статусом `Superseded by ADR-NNNN`. История решений сохраняется.

**Статусы ADR:** `Proposed` / `Accepted` / `Deprecated` / `Superseded`.

### 4.3. Политики авторизации — `POL-<NNN>`

Используются на Этапе 2+ при появлении RBAC. Описывают конкретное правило доступа: «кто может что делать с каким ресурсом».

**Формат ID:** `POL-<NNN>`.

**Имя файла:** `POL-<NNN>-<kebab-case-title>.md`.

**Примеры:**

```
POL-001-dish-ownership.md
POL-002-media-access.md
POL-003-media-ownership.md
```

В Use Cases ссылка на политику идёт через ID: `Authorization: POL-001`.

### 4.4. Доменные события — `EVT-<MODULE>-<NNN>`

Идентификатор для каждого доменного события системы. Полезно при росте количества событий и подписчиков.

**Формат ID:** `EVT-<MODULE>-<NNN>`.

**Имя файла:** `EVT-<MODULE>-<NNN>-<EventName>.md` (если описание событий выносится в отдельные документы; на старте — может быть таблица в `events.md` модуля).

**Примеры:**

```
EVT-AUT-001-UserRegistered
EVT-DSH-001-DishPublished
EVT-DSH-002-DishRated
EVT-MED-001-MediaUploaded
```

### 4.5. Бизнес-правила и инварианты — `BR-<MODULE>-<NNN>`

Опционально. Применять при значительном количестве доменных правил, требующих перекрёстных ссылок из UC.

**Формат:** `BR-<MODULE>-<NNN>`.

**Примеры:**

```
BR-DSH-001-DishPublishRequiresMainImage
BR-DSH-002-DishMaxThreeCategories
BR-MED-001-PersonalDataRequiresOwner
```

### 4.6. Сводная таблица типов идентификаторов

| Тип артефакта | Префикс | Формат | Пример |
|---------------|---------|--------|--------|
| Use Case | `UC` | `UC-<MOD>-<NNN>` | `UC-DSH-001` |
| Architecture Decision Record | `ADR` | `ADR-<NNNN>` | `ADR-0001` |
| Authorization Policy | `POL` | `POL-<NNN>` | `POL-001` |
| Domain Event | `EVT` | `EVT-<MOD>-<NNN>` | `EVT-DSH-001` |
| Business Rule | `BR` | `BR-<MOD>-<NNN>` | `BR-DSH-001` |

## 5. Шаблоны документов

Все шаблоны лежат в `docs/public/DocTemplates/`. На текущий момент:

- `UseCase-template.md` — шаблон для описания Use Case.

**В будущем стоит добавить:**

- `ADR-template.md` — шаблон ADR (по классической схеме: Context → Decision → Consequences).
- `DomainModel-template.md` — шаблон описания доменной модели модуля (на основе текущих `Dishes/domain-model.md` и `Media/domain-model.md`).
- `Policy-template.md` — шаблон описания политики авторизации.

При создании нового документа — копируем шаблон, переименовываем согласно правилам выше, заполняем.

### 5.1. Удобный формат ADR

```markdown
# ADR-NNNN: Название решения

**Status:** Accepted
**Date:** YYYY-MM-DD

## Context
В каких условиях принимается решение, какая проблема решается.

## Decision
Что решили.

## Consequences
- Положительные следствия
- Отрицательные следствия / компромиссы
- Что становится сложнее
```

ADR пишется кратко (1–2 страницы), не нужно растекаться.

## 6. Шпаргалка-выжимка

Чтобы не перечитывать всё каждый раз — короткая выжимка:

- **Wiki** — обзорные документы для внешних читателей, кириллица, цифровые префиксы.
- **`docs/public/`** — рабочая техническая документация, латиница, kebab-case.
- **`docs/public/modules/<name>/`** — доменная модель + use cases по модулю.
- **`docs/public/adr/`** — архитектурные решения уровня проекта.
- **`docs/public/policies/`** — политики авторизации.
- **`docs/public/DocTemplates/`** — шаблоны для новых документов.
- **ID:** UC-MOD-NNN, ADR-NNNN, POL-NNN, EVT-MOD-NNN, BR-MOD-NNN.
- **Без этапов в именах файлов** — версия и статус живут в шапке документа.
- **README.md в каждой папке** создаём по мере необходимости.
