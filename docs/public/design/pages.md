# Страницы веб-интерфейса — реестр и приоритеты

> **Версия:** 1.0
> **Последнее изменение:** 2026-07-21
> **Статус:** Приоритет 1 — принят; приоритеты 2–3 — предварительный список, уточняется по ходу Этапа 4
> **Связанные документы:** [brand-book.md](brand-book.md) (дизайн-система, §7 — дизайн страниц), [ADR-0018](../adr/ADR-0018-web-frontend-stack.md) (стек: React + TypeScript SPA), [реестр UC Dishes](../modules/dishes/use-cases/README.md), Wiki [[05_Дорожная-карта]] §Этап 4

Документ фиксирует состав страниц веб-интерфейса Этапа 4, их привязку к фактическому API и порядок реализации. Брифы на макеты страниц готовятся отдельно, по одному на страницу, по мере подхода очереди.

---

## 1. Принципы

1. **Дизайн подстраивается под контракт API, а не наоборот.** Backend закрыт и стабилен; страница проектируется от того, что отдают эндпоинты. Потребности интерфейса, требующие доработки API, не блокируют страницу — они либо реализуются маленьким слайсом вместе с ней (фронт + бэк), либо попадают в бэклог (§6).
2. **Границы MVP.** Без модуля Orders (Этап 6) — нет корзины, цен заказа, кнопок «Заказать», оформления заказа. Без модуля Social (Этап 5) — нет выставления оценок, отзывов, комментариев, избранного. Рейтинг и счётчики просмотров/избранного **показываются** (данные есть в API), но действий с ними нет.
3. **Сквозной сценарий приоритета 1:** посмотреть каталог → открыть блюдо → зарегистрироваться → купить подписку → получить полный рецепт с калькулятором порций. Демонстрирует все пять модулей backend.
4. **Служебные состояния** — не отдельные страницы, но обязательные состояния каждой: загрузка (скелетоны с «паром» — бренд-бук §10), пустой результат, ошибка с указанием действия, 404. Прорабатываются в брифе каждой страницы.
5. **Механика аутентификации** (хранение JWT, тихое обновление по `POST /api/auth/refresh`, редиректы на вход) — сквозная реализационная задача SPA, не страница; в брифы не входит.

## 2. Приоритет 1 — сквозной сценарий (реализуются первыми)

### 2.1. Главная

- **Назначение:** входная точка; аппетит + понимание, что это за платформа; переход в каталог.
- **API:** `GET /api/dishes/search` (`sortBy=Newest` — «новинки», `ViewsDesc` — «популярное», `RatingDesc`), `GET /api/categories/tree` (тематические входы в каталог), `GET /api/tags/popular` (чипсы популярных запросов у поиска).
- **Примечания:** блок «повара недели» из бренд-бука §7 не реализуем — в API нет списка поваров (§6). Hero, строка поиска, лента карточек — по бренд-буку.

### 2.2. Каталог

- **Назначение:** поиск и фильтрация опубликованных блюд (UC-DSH-054).
- **API:** `GET /api/dishes/search` — `text`, `categoryIds[]`, `tagIds[]`, `dietLabelsMask`, `difficulties[]`, `costs[]`, `minRating`, `sortBy`, `page`, `pageSize` (≤ 25); `GET /api/categories/tree`, `GET /api/tags/popular` — источники фильтров. Миниатюры — `GET /api/media/{id}/thumbnail`.
- **Состав фильтров — строго по контракту:** категории, теги, диетические метки, сложность, стоимость, минимальный рейтинг. Фильтров по времени приготовления, КБЖУ и аллергенам в API нет — в интерфейс не закладываются (расширение — §6).
- **Карточка выдачи — по фактическому `DishCardListItemDto`:** фото-«тарелка», название, краткое описание, сложность · стоимость · рейтинг (золотые звёзды + счётчик), бейджи диет. Времени приготовления и числа порций в DTO нет — строка метаданных бренд-бука «время · порции · сложность» сокращается до доступного (расширение DTO — §6).
- **Пагинация:** кнопка «Показать ещё» (бренд-бук §7).

### 2.3. Карточка блюда

- **Назначение:** публичная витрина блюда + точка входа в Premium-гейт.
- **API:** `GET /api/dishes/by-slug/{slug}` (анонимно, `DishDetailDto`), автор — `GET /api/users/{id}` (анонимно), фото — `GET /api/media/{id}`; счётчик просмотров — `POST /api/dishes/{id}/views`.
- **Блок рецепта — три состояния:** аноним → приглашение войти (401); авторизованный без гранта → предложение Premium с переходом на тарифы (403 `DISHES.PREMIUM_REQUIRED`); с грантом → полный рецепт `GET /api/dishes/{id}/recipe` + калькулятор порций `GET /api/dishes/{id}/recipe/scaled?servings=` (степпер, tabular-nums).
- **Примечания:** рейтинг показывается, но не ставится (Social — Этап 5); кнопки «Заказать у повара» нет (Orders — Этап 6); сердце избранного — неактивно или скрыто (§6).

### 2.4. Вход / Регистрация

- **Назначение:** аутентификация; регистрация ведёт к покупке подписки в сквозном сценарии.
- **API:** `POST /api/auth/register`, `POST /api/auth/login`; `refresh` и `logout` — механика (§1.5).
- **Примечания:** формы одной колонкой, max 560 px (бренд-бук §7); тексты ошибок — «что случилось + что делать» (ToV §2).

### 2.5. Тарифы — витрина «Специи»

- **Назначение:** витрина планов подписки; визуальная система «Специи» (бренд-бук §3.7) на фактических данных.
- **API:** `GET /api/subscription-plans` (UC-SUB-040) — `PublicName`, `Description`, `PlanKind` (Base/AddOn), `RequiredRole` (пометка «требуется подтверждение статуса»), состав услуг `Grants`, офферы `Offers` (период, цена, валюта).
- **Примечания:** карточки «Соль / Шафран / Трюфель» связываются с планами по данным витрины, а не хардкодом; бейдж текущего тарифа — если пользователь авторизован и подписка известна (см. 2.7).

### 2.6. Оформление подписки

- **Назначение:** покупка выбранного оффера; зона оплаты в Trust-палитре (бренд-бук §3.4).
- **API:** `POST /api/user-subscriptions` (UC-SUB-020) — `PriceId`, `PaymentMethodId` (mock-токен шлюза-заглушки, см. ADR-0017), `AcceptedTermsAt`.
- **Обязательное в UI:** чекбокс согласия с офертой (контракт требует момент согласия); обработка ошибок `409 SUBS.ALREADY_HAS_BASE` (уже есть базовая подписка), `403 SUBS.FORBIDDEN_ROLE_REQUIRED` (роль-гейт плана), `400 SUBS.OFFER_NOT_PURCHASABLE`.
- **Примечания:** страница или шаг-панель с витрины — решается в брифе; после успеха — переход к рецепту или в кабинет (закрытие сквозного сценария).

### 2.7. Личный кабинет

- **Назначение:** профиль + управление подпиской.
- **API (профиль):** `GET /api/users/me`, `GET /api/users/me/roles`; редактирование — `PUT me/personal-info`, `me/location`, `me/visibility`, `me/email`, `me/phone`, `me/username`; аватар — `POST /api/media/upload` + `PUT me/avatar`.
- **API (подписка):** `GET /api/user-subscriptions/{id}` (UC-SUB-021) — статус, период, `CancelAtPeriodEnd`; отмена — `POST /api/user-subscriptions/{id}/cancel` (доступ до конца оплаченного периода — показать дату).
- **Требует доработки API:** эндпоинт «моя текущая подписка» (`GET /api/user-subscriptions/my`) — сейчас идентификатор подписки известен только из ответа оформления; после перелогина раздел «Подписка» построить не из чего. Реализуется вместе с этой страницей (фронт + бэк), см. §6.
- **Примечания:** разделы меню кабинета — только реальные: профиль, подписка (+ «мои блюда» при реализации приоритета 2). «Избранное» и «заказы» из бренд-бука §7 — будущие этапы.

## 3. Приоритет 2 — средний (после закрытия сквозного сценария)

### 3.1. Страница автора

- **Назначение:** публичная витрина блюд одного автора — «у каждого блюда есть автор, история и лицо». Упрощённая версия: без заказов, отзывов и persональной обложки (это «карточка повара» Этапа 6).
- **API:** `GET /api/users/{id}` (анонимно) + `GET /api/dishes/by-author/{authorUserId}` (анонимно, пагинация). API готово полностью — страница дешёвая.

### 3.2. Мои блюда (черновики и публикации)

- **Назначение:** входная точка авторского флоу — списки своих черновиков и опубликованных блюд.
- **API:** `GET /api/dishes/my-drafts` (UC-DSH-053), `GET /api/dishes/by-author/{myId}` (публикации); действия из списка: `POST {id}/publish`, `POST {id}/unpublish`, `POST {id}/archive`.

### 3.3. Редактор блюда — карточка

- **Назначение:** создание черновика и редактирование карточки: основные поля, категории, теги, диетические метки, история, главное фото.
- **API:** `POST /api/dishes` (создание), `PATCH {id}` (поля карточки), `PATCH {id}/diet-labels`, `PATCH {id}/history`, `PATCH {id}/main-image` (+ `POST /api/media/upload`), `PUT {id}/categories`, `PUT {id}/tags`; публикация — `POST {id}/publish`.
- **Примечания:** публикация проходит доменные проверки полноты (UC-DSH-004) — новое блюдо, созданное только через редактор карточки, может не пройти публикацию без заполненного рецепта. Полный авторский сценарий «создал → опубликовал» замыкается редактором рецепта (§4.1); до его появления рецепт заполняется через Swagger.

## 4. Приоритет 3 — низкий (по остатку времени)

### 4.1. Редактор рецепта

- **Назначение:** ингредиенты (каталожные и freeform), шаги, тайминг, выход/порции, КБЖУ.
- **API:** ~15 эндпоинтов `PUT`/`POST`/`DELETE` под `/api/dishes/{id}/recipe/**` + справочники `GET /api/ingredients/search`, `GET /api/measure-units`.
- **Примечания:** самая тяжёлая страница интерфейса; реализуется последней из авторского флоу.

### 4.2. Админ-справочники

- **Назначение:** управление категориями, тегами, ингредиентами (админские UC 100+).
- **API:** CRUD `/api/categories`, верификация/удаление `/api/tags`, CRUD + деактивация `/api/ingredients`.
- **Примечания:** для защиты диплома операции доступны через Swagger UI — страница оправдана только при остатке времени; ролевой гейт — Admin.

## 5. Вне периметра Этапа 4

Описаны в бренд-буке §7 как задел, в интерфейс Этапа 4 не входят:

| Страница / блок | Причина |
|---|---|
| Корзина, оформление заказа | Модуль Orders — Этап 6 |
| Карточка повара (полная: заказы, «Написать», обложка) | Orders (Этап 6) + Social (Этап 5); упрощённая витрина — §3.1 |
| Отзывы, комментарии, выставление оценок | Модуль Social — Этап 5 |
| Избранное (действие и раздел кабинета) | Модуль Social — Этап 5; счётчики показываются |
| Живые подсказки в поиске (блюда/категории/повара) | Нет эндпоинта подсказок; полнотекстовый поиск — Этап 8+ |

## 6. Бэклог доработок API, порождённый интерфейсом

Потребности страниц, не закрытые текущим контрактом. Не блокируют реализацию (страницы проектируются от текущего API), кроме первого пункта:

| # | Доработка | Когда |
|---|---|---|
| 1 | **`GET /api/user-subscriptions/my`** — текущая подписка пользователя без знания её id | Вместе со страницей кабинета (§2.7), фронт + бэк |
| 2 | **Время, порции и уровень доступа в `DishCardListItemDto`** — для строки метаданных карточки «время · порции · сложность» (бренд-бук §7) и бейджа тарифа-«специи» на карточке; время и порции есть в снепшоте (`RecipeTiming`, `RecipeYield`), нужна проекция; пер-блюдная маркировка Premium — открытый вопрос модели доступа | По мере возможности, после приоритета 1 |
| 3 | **Фильтр по времени приготовления** (`maxTotalTimeMinutes`) — отложен ещё на Этапе 2 (реестр UC Dishes, §UC-DSH-054) | Вместе с п. 2 |
| 4 | **Фильтры по аллергенам и КБЖУ** в поиске | Обсуждается; полнотекстовый поиск и расширенная фильтрация — кандидат на Elasticsearch/OpenSearch, Этап 8+ (отдельный ADR) |
| 5 | **Эндпоинт подсказок поиска** (typeahead) | Этап 8+, вместе с п. 4 |
| 6 | **Избранное** (добавление/список) | Модуль Social — Этап 5 |
| 7 | **Список поваров** («повара недели» на главной) | Не раньше Social/Orders — требует витринных данных о поварах |
| 8 | **Batch-получение ингредиентов по списку id** — рецепт отдаёт `IngredientId` без имени; сейчас страница рецепта делает N параллельных `GET /api/ingredients/{id}` | По мере возможности; терпимо для MVP (рецепты ≤ пары десятков позиций) |
| 9 | **Галерея фото блюда** — публичного эндпоинта списка медиа блюда нет, в `DishDetailDto` только `MainImageId`; бренд-бук §7 описывает галерею с миниатюрами | Обсуждается; в MVP — одно главное фото |
